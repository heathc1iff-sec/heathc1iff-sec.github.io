# 域前置现实限制说明

## 这篇文档解决什么问题

课程里的域前置用于解释一种网络过滤绕过思路：让外层网络设备看到高信誉前置域名，而 HTTP 层请求被路由到真实后端。

但现实里，许多大型云厂商和 CDN 已经限制或关闭了传统域前置能力。初学者复习时容易困惑：

- 课程为什么讲这个？
- 为什么我照着配置不成功？
- SNI、Host、证书、CDN 回源到底哪个出错？
- 考试要掌握到什么程度？

这篇文档把“课程知识点”和“现实限制”分开，避免把失败都归咎于 payload 或 C2。

---

## 域前置解决什么问题

企业网络常按域名信誉、分类、SNI、Host、证书来判断是否允许出站流量。

域前置的核心想法：

```text
TLS 层看起来访问 front.example-cdn.com
HTTP Host 指向 real-backend.example.com
CDN 根据 Host 把请求转到真实后端
```

目标是让防火墙/代理看到一个允许访问的前置域名，而真实 C2 或后端隐藏在 CDN 路由之后。

---

## SNI、Host、证书、回源的关系

| 层 | 字段/对象 | 作用 |
|---|---|---|
| TLS | SNI | TLS 握手时告诉服务器想访问哪个域名 |
| TLS | 证书 | 证明服务端身份，通常要匹配 SNI |
| HTTP | Host Header | HTTP 请求层告诉 Web/CDN 要访问哪个站点 |
| CDN | 回源规则 | 根据 Host/路径等规则把请求转发到后端 |

传统域前置依赖 SNI 和 Host 不一致仍能被 CDN 接受。

现代很多平台会强制：

- SNI 必须和 Host 一致或属于同一租户。
- Host 必须是该 CDN 配置中已绑定的域名。
- 证书、租户、分发配置必须匹配。
- 不允许跨租户回源。

所以“理论上懂”和“当前平台可用”是两件事。

---

## 初学者最容易混淆的点

| 误解 | 正确认识 |
|---|---|
| 域前置就是把 Host 改掉 | Host 只是其中一层，还要看 SNI、证书和 CDN 路由 |
| CDN 都支持域前置 | 很多现代 CDN 已限制传统方式 |
| 失败一定是 C2 配置错 | 可能是平台策略不允许 |
| HTTPS 成功就说明前置成功 | 可能只是访问了前置域名，没有到真实后端 |
| 证书报错可以忽略 | 考试排错时证书错误是重要线索 |

## 判断一个配置是否可能成立

### 1. 前置域名是否能访问

```powershell
Test-NetConnection front.example.com -Port 443
curl -I https://front.example.com
```

如果前置域名本身都不能访问，先查网络过滤、DNS、代理。

### 2. SNI 是否正确

多数工具默认用 URL 里的域名作为 SNI。若你手动连 IP 或自定义 Host，要确认 TLS 层 SNI 是谁。

常见问题：

| 现象 | 可能原因 |
|---|---|
| TLS 握手失败 | SNI 不被接受、TLS 版本、证书问题 |
| 证书域名不匹配 | SNI/目标不一致 |
| 连接成功但后端无请求 | Host/CDN 路由未到真实后端 |

### 3. Host 是否被 CDN 接受

```bash
curl -k -H "Host: real-backend.example.com" https://front.example.com/
```

观察：

- 返回是否来自真实后端。
- CDN 是否返回 403/404/421。
- 后端日志是否有请求。

### 4. 后端是否绑定到同一 CDN/租户

如果真实后端没有在该 CDN 配置中绑定，CDN 通常不会帮你转发。

这不是命令能绕过的问题，而是平台策略问题。

## 域前置失败排错表

| 现象 | 更可能原因 | 下一步 |
|---|---|---|
| DNS 解析失败 | 前置域名被拦或 DNS 问题 | 查 `Resolve-DnsName`、内部 DNS |
| 443 端口不通 | 出站端口或 IP 被拦 | 查 `Test-NetConnection` |
| TLS 失败 | SNI、证书、TLS inspection | 查证书、代理、SNI |
| 返回 CDN 403/404 | Host 未绑定或跨租户被拒 | 查 CDN 配置 |
| 前置域名成功但 C2 无请求 | Host/路径/回源规则错 | 看后端日志 |
| 浏览器成功但 payload 失败 | payload 不支持代理/SNI/Host 配置 | 换代理感知通信或工具参数 |
| 曾经可用后来失效 | CDN 政策或租户配置变化 | 记录现实限制，换路线 |

## 与其他出网路线的取舍

| 场景 | 优先考虑 |
|---|---|
| 浏览器能出，payload 不出 | 代理感知通信、WinHTTP/HKCU 代理 |
| 只有少数域名能访问 | 高信誉允许域、合法证书、路径伪装 |
| HTTP/HTTPS 被严格限制但 DNS 可用 | DNS 隧道 |
| TLS 被检查且证书异常 | 合法证书、企业根证书影响、换协议 |
| CDN 域前置失败 | 记录 SNI/Host/CDN 证据，切回代理或 DNS |

## 课程知识点应该掌握到什么程度

对 OSEP 备考，域前置的重点是理解和排错，而不是依赖某个云厂商永远可用。

你要能解释：

1. SNI 和 Host 的区别。
2. 防火墙/代理在 TLS 层和 HTTP 层分别能看到什么。
3. 为什么 CDN 可以根据 Host 做路由。
4. 为什么现代平台会限制传统域前置。
5. 失败时如何判断是 DNS、TCP、TLS、Host、CDN、C2 哪一层。

## OSEP 相关重点

1. 域前置是网络过滤绕过的思维模型，不要把它当成一定可用的固定命令。
2. SNI、证书、Host、CDN 回源四层必须分开排错。
3. 现代限制本身就是排错结论，可以写入笔记，不要无限重试。
4. 如果是 Office/JScript 回连失败，要先用 `10-网络过滤绕过/09-网络过滤新手详解与出网排错.md` 分层判断。
5. 能解释为什么失败，比背一个过时 CDN 配置更重要。
