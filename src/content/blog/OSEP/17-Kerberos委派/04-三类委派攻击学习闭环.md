# 三类 Kerberos 委派攻击学习闭环

## 委派到底解决什么问题

委派允许一个服务代表用户去访问另一个服务。配置不当时，攻击者可以滥用这种“代表访问”的能力获取高权限服务票据。

---

## Kerberos 基础回顾

| 概念 | 说明 |
|---|---|
| TGT | 用户向 KDC 证明身份后的主票据 |
| TGS | 访问具体服务的票据 |
| SPN | 服务主体名，表示服务身份 |
| S4U2Self | 服务为某个用户向自己请求票据 |
| S4U2Proxy | 服务代表用户请求另一个服务票据 |
| altservice | 改服务类型以访问不同服务 |

---

## 三类委派对比

| 类型 | 关键字段 | 风险 |
|---|---|---|
| 无约束委派 | `TRUSTED_FOR_DELEGATION` | 可捕获转发来的 TGT |
| 约束委派 | `msDS-AllowedToDelegateTo` | 可代表用户访问指定服务 |
| RBCD | `msDS-AllowedToActOnBehalfOfOtherIdentity` | 资源端允许某主体代表用户访问自己 |

---

## 看到枚举结果怎么判断

| 发现 | 意义 |
|---|---|
| 主机无约束委派 | 如果控制该主机，可能等高权限用户连接 |
| 服务账户有约束委派 | 如果控制服务账户，可请求指定 SPN 的票据 |
| 机器对象可写 | 可能写 RBCD |
| MachineAccountQuota > 0 | 普通用户可能创建机器账户 |
| 有高权限用户访问路径 | 可联动打印机强制认证或服务访问 |

---

## 场景化打法

### 控制无约束委派主机

```
控制主机
  -> 等待或强制高权限认证
  -> 捕获 TGT
  -> 导入票据
  -> 访问目标服务
```

### 拿到约束委派服务账户

```
控制服务账户
  -> 查看允许委派到的 SPN
  -> S4U2Self
  -> S4U2Proxy
  -> 使用服务票据访问目标
```

### 有机器对象写权限

```
准备可控机器账户
  -> 写目标机器 RBCD
  -> 请求目标服务票据
  -> 访问 CIFS/LDAP/HOST
```

---

## 常见失败与排错

| 现象 | 排查 |
|---|---|
| 请求票据失败 | SPN 错误、时间不同步、账户权限 |
| 票据导入后不能访问 | 服务名不对、altservice、目标权限 |
| RBCD 写入失败 | 机器对象权限不足 |
| 创建机器账户失败 | MachineAccountQuota 为 0 或权限限制 |
| CIFS 可用但 LDAP 不行 | 服务票据类型不匹配 |

---

## OSEP 重点

1. 能区分无约束、约束、RBCD。
2. 能解释 S4U2Self 和 S4U2Proxy。
3. 能把“控制服务账户/机器对象”转成票据请求链。
4. 能排查 SPN、时间、服务名、票据导入问题。
