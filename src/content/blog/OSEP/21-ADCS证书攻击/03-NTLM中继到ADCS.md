# NTLM 中继到 ADCS

## 核心思路

NTLM Relay 到 ADCS 的本质是：把某个高价值主体发起的 NTLM 认证转发到 ADCS HTTP 端点，借这个主体的身份向 CA 申请证书。证书一旦能用于认证，就可能把一次短暂的 NTLM 认证变成长期可复用的访问能力。

课程里的重点不是“背一条命令”，而是判断这条链是否成立。

---

## 成立条件

| 条件 | 判断问题 |
|---|---|
| 存在 ADCS HTTP 端点 | 是否能访问 Web Enrollment、CES、CEP 等端点 |
| 端点接受 NTLM | 是否能被中继，是否存在 EPA/签名/通道绑定限制 |
| 能触发目标认证 | 是否能让目标机器或用户向你控制的位置发起 NTLM |
| 模板可被目标主体申请 | 机器账户通常需要机器模板，用户账户需要用户可用模板 |
| 申请出的证书可用于后续认证 | 是否能走 PKINIT 或 Schannel |

---

## 判断流程

```
枚举 CA 和 HTTP 端点
        |
        v
确认端点是否接受 NTLM
        |
        v
选择要中继的身份：机器账户还是用户账户
        |
        v
确认该身份可申请的模板
        |
        v
触发认证并申请证书
        |
        v
把证书转换为访问能力
        |
        v
继续 LDAP 修改、横向移动或凭证获取
```

---

## 机器证书路径的后续思考

如果中继拿到的是机器证书，优先思考：

| 问题 | 可能方向 |
|---|---|
| 这台机器是否是 DC | 机器身份可能具备高价值目录访问 |
| 这台机器是否能写某些对象 | 看 RBCD、ACL、服务访问 |
| 这台机器是否能访问敏感共享 | 读取配置、备份、部署脚本 |
| 能否用机器身份获取服务票据 | 尝试 Kerberos 路线 |
| 是否能进一步触发到 DC 机器账户 | 继续找 coercion 机会 |

---

## 常见失败原因

| 现象 | 优先排查 |
|---|---|
| HTTP 端点不可访问 | 端口、虚拟目录、代理、主机名、TLS |
| 中继失败 | EPA、SMB 签名、通道绑定、目标只走 Kerberos |
| 证书申请失败 | 模板名、目标身份是否有注册权限、CA 名称 |
| 证书无法认证 | PKINIT 支持、EKU、证书链、时间同步 |
| 认证成功但权限不足 | 证书身份不是你以为的身份，或目标服务不接受该身份 |

## 证据记录

报告里不要只写“利用 ADCS 成功”。建议记录：

1. CA 主机和 CA 名称。
2. HTTP 端点路径和访问方式。
3. 被中继主体的身份。
4. 申请的模板名称。
5. 证书对应的身份。
6. 证书转换成访问能力后的验证结果。
7. 后续造成的权限影响，例如读取敏感文件、写入目录对象、横向移动或提权。
