# 编程语言与执行模型完整指南

> 对应参考资料：`1：office钓鱼.html` 前置理论、`4：反射式powershell.html`、`6：进程注入迁移挖空.html` 中反复出现的编程基础。  
> 学习定位：这不是为了把你训练成软件工程师，而是让你能读懂 OSEP 里 C#、PowerShell、VBA、JScript、Win32 API、反射和 shellcode 的共同语言。

## 这篇解决什么问题

初学 OSEP 时，很多概念会一起砸过来：

```text
C#、PowerShell、VBA、JScript
IL、CLR、JIT、Assembly
P/Invoke、DllImport、Marshal.Copy
x86、x64、opcode、shellcode
托管代码、非托管代码、反射
```

如果这些基础没有串起来，后面每章都会像孤岛。这篇文档的目标是建立一张“代码如何变成执行”的地图。

你读完后应该能回答：

```text
1. 不同语言为什么执行方式不同？
2. shellcode 为什么必须匹配 CPU/进程架构？
3. C# 为什么叫托管代码？
4. C# 和 PowerShell 为什么还能调用 Win32 API？
5. 反射为什么能动态加载和调用代码？
6. 这些基础分别对应 OSEP 哪些章节？
```

## 一、先看 OSEP 中常见语言

| 语言 | 常见位置 | 主要用途 | 学习重点 |
|---|---|---|---|
| VBA | Office 宏攻击 | 宏入口、调用 Win32 API、启动 PowerShell | `Declare`、`PtrSafe`、`LongPtr` |
| JScript/VBScript | WSH 钓鱼、DotNetToJScript | 客户端脚本执行、COM 调用 | `ActiveXObject`、COM、脚本宿主 |
| PowerShell | 反射加载、后渗透、枚举 | 调用 .NET、下载执行、内存加载 | AMSI、CLM、反射、管道 |
| C# | 注入器、加载器、工具改造 | P/Invoke、进程注入、反射 DLL | `DllImport`、`IntPtr`、`Marshal` |
| C/C++ | 底层原理、非托管工具 | 原生 API、内存、线程 | 指针、结构体、编译 |
| Python/Bash | 辅助自动化 | 快速脚本、服务、处理数据 | 实验辅助，不是 Windows 内存执行主线 |

初学顺序建议：

```text
先学概念，不急着写完整工具
  -> 能看懂 C# 方法和类
  -> 能看懂 PowerShell 调 .NET
  -> 能看懂 Win32 API 参数
  -> 再进入 Office、反射、进程注入
```

## 二、编译型、解释型与中间语言

### 2.1 编译型语言

编译型语言先把源代码转换成机器可以执行的形式，再运行。

```text
源代码
  -> 编译器
  -> 可执行文件或库
  -> CPU 执行
```

代表语言：

| 语言 | 输出 | 特点 |
|---|---|---|
| C | 原生机器码 | 最接近系统，常见于底层组件 |
| C++ | 原生机器码 | Windows 大量组件和工具使用 |
| Go/Rust | 原生二进制 | 常用于独立工具 |

渗透测试意义：

| 场景 | 为什么需要编译型语言 |
|---|---|
| 写注入器 | 需要调用底层 API |
| 写加载器 | 需要处理内存、线程、指针 |
| 绕过检测 | 可以控制编译产物和行为模式 |
| 分发工具 | 编译后可独立运行 |

### 2.2 解释型语言

解释型语言由解释器读取代码并执行。

```text
脚本文件
  -> 解释器
  -> 逐步解析/执行
```

代表语言：

| 语言 | 解释器/宿主 | 常见用途 |
|---|---|---|
| Python | `python.exe` | PoC、数据处理、临时 Web 服务 |
| JavaScript/JScript | 浏览器或 WSH | 客户端脚本、钓鱼入口 |
| PowerShell | `powershell.exe` | Windows 后渗透、反射加载 |
| Bash | `/bin/bash` | Linux 自动化 |

优点：

```text
开发快
修改快
适合实验和自动化
```

缺点：

```text
依赖解释器
源码容易被分析
容易被 AMSI/脚本日志/EDR 观察
```

### 2.3 中间语言：Java 和 .NET 的折中模型

C# 不是简单地“直接变成机器码”。它通常先编译成 IL，再由 CLR 在运行时处理。

```text
C# 源代码
  -> C# 编译器
  -> IL / CIL / MSIL
  -> CLR 加载
  -> JIT 编译为机器码
  -> CPU 执行
```

Java 类似：

```text
Java 源代码
  -> bytecode
  -> JVM
  -> JIT/解释执行
```

对比：

| 项目 | C/C++ | C#/.NET | PowerShell |
|---|---|---|---|
| 初始代码 | 源代码 | 源代码 | 脚本 |
| 中间形式 | 通常无 | IL | AST/脚本块/.NET 对象 |
| 运行时 | 操作系统直接加载 | CLR | PowerShell + CLR |
| 内存管理 | 程序员负责 | CLR/GC 负责 | CLR/PowerShell 负责 |
| 调 Win32 API | 直接调用或链接 | P/Invoke | Add-Type/反射/PInvoke |

## 三、从高级代码到 CPU 指令

### 3.1 高级语言、汇编和机器码

你可以把程序执行看成逐层下降：

```text
C# / C / PowerShell / VBA
  -> 编译器或解释器处理
  -> 汇编或中间表示
  -> 机器码
  -> CPU 指令
```

机器码是 CPU 真正执行的字节序列。shellcode 本质上就是一段精心组织的机器码。

### 3.2 Opcode 是什么

Opcode 是 CPU 指令的二进制编码。

| 十六进制 | 汇编 | 常见含义 |
|---|---|---|
| `0x90` | `NOP` | 空操作 |
| `0xC3` | `RET` | 返回 |
| `0xCC` | `INT 3` | 断点 |
| `0xE8` | `CALL` | 调用 |

Shellcode 示例经常是这样的：

```text
fc 48 83 e4 f0 e8 c0 00 00 00 ...
```

这些不是普通字符串，而是 CPU 能解释的指令字节。

### 3.3 CPU 架构决定 shellcode 是否能运行

不同 CPU 架构有不同指令集。

| 架构 | 说明 | OSEP 关注 |
|---|---|---|
| x86 | 32 位 Intel/AMD 指令集 | 仍会遇到，尤其 32 位 Office/进程 |
| x64 / AMD64 | 64 位 Windows 主流架构 | 非常重要 |
| ARM | 移动和部分新设备 | OSEP 主线较少 |

重点：

```text
x86 shellcode 不能当作 x64 shellcode 使用
x64 shellcode 不能注入 32 位进程
Office 位数决定 VBA 进程位数
PowerShell 位数决定 PowerShell runner 使用的 payload 位数
```

这就是为什么后面经常先查：

```powershell
[Environment]::Is64BitOperatingSystem
[Environment]::Is64BitProcess
$env:PROCESSOR_ARCHITECTURE
```

## 四、托管代码与非托管代码

### 4.1 非托管代码

非托管代码通常直接编译为机器码，由程序员自己管理资源。

代表：

```text
C
C++
原生 DLL
大部分 Win32 API 所在系统组件
```

特点：

| 特点 | 说明 |
|---|---|
| 直接接近系统 | 可以直接操作指针、内存、句柄 |
| 性能高 | 少一层运行时抽象 |
| 风险高 | 内存泄漏、越界、崩溃更常见 |
| 攻击面大 | 经典内存漏洞多来自非托管代码 |

C 语言内存示例：

```c
char* buffer = (char*)malloc(100);
strcpy(buffer, "hello");
free(buffer);
```

如果忘记 `free`，可能内存泄漏；如果写超过 100 字节，可能越界。

### 4.2 托管代码

托管代码由运行时管理内存和执行环境。

代表：

```text
C#
VB.NET
PowerShell 使用的大量 .NET 对象
Java
```

特点：

| 特点 | 说明 |
|---|---|
| 自动内存管理 | 垃圾回收器负责释放对象 |
| 类型信息丰富 | 程序集包含元数据 |
| 易于反射 | 可以动态发现类型、方法、字段 |
| 受运行时约束 | 某些底层操作需要借助 P/Invoke |

### 4.3 OSEP 为什么一直在托管/非托管之间切换

OSEP 很多技术就是“用托管代码控制非托管能力”。

```text
C# / PowerShell
  -> 方便编写、动态加载、反射
  -> 通过 P/Invoke 找到 kernel32.dll
  -> 调用 VirtualAlloc / CreateThread 等非托管 API
  -> 完成内存执行或进程注入
```

这条桥就是 P/Invoke。

## 五、.NET、IL、CLR、JIT

### 5.1 .NET Framework 是什么

.NET Framework 是 Windows 上常见的软件运行框架。它包含：

| 组件 | 说明 |
|---|---|
| CLR | Common Language Runtime，公共语言运行时 |
| FCL/BCL | 类库，提供文件、网络、加密、反射等能力 |
| IL | Intermediate Language，中间语言 |
| JIT | Just-In-Time，将 IL 编译为机器码 |

### 5.2 C# 程序执行流程

```text
Program.cs
  -> csc.exe 编译
  -> Program.exe，内部包含 IL 和元数据
  -> CLR 加载程序集
  -> JIT 编译被调用的方法
  -> CPU 执行机器码
```

### 5.3 程序集 Assembly

程序集是 .NET 的部署单位，通常是 `.exe` 或 `.dll`。

它包含：

```text
IL 代码
类型元数据
方法元数据
资源
清单 Manifest
```

为什么这对 OSEP 重要？

| 技术 | 与程序集的关系 |
|---|---|
| 反射加载 | 从文件或字节数组加载程序集 |
| 内存执行 | `Assembly.Load(byte[])` 不落地加载 |
| DotNetToJScript | 把 .NET 程序包装进 JScript |
| 反射式 PowerShell | 动态发现类型和方法 |

### 5.4 反射 Reflection

反射允许程序在运行时观察和操作自身或其他程序集的类型。

常见能力：

```csharp
Assembly asm = Assembly.Load(byteArray);
Type type = asm.GetType("Namespace.ClassName");
object obj = Activator.CreateInstance(type);
MethodInfo method = type.GetMethod("Run");
method.Invoke(obj, new object[] { });
```

你现在不需要背 API。你只要理解：

```text
反射 = 运行时看见类型和方法
      + 动态创建对象
      + 动态调用方法
```

后面 `05-反射式PowerShell` 会把这件事讲到攻击链里。

## 六、P/Invoke：托管代码调用 Win32 API 的桥

### 6.1 P/Invoke 是什么

P/Invoke 是 Platform Invocation Services，允许 .NET 托管代码调用非托管 DLL 函数。

基本形式：

```csharp
using System;
using System.Runtime.InteropServices;

class Win32
{
    [DllImport("user32.dll", CharSet = CharSet.Ansi)]
    public static extern int MessageBoxA(
        IntPtr hWnd,
        string text,
        string caption,
        uint type
    );
}
```

调用：

```csharp
Win32.MessageBoxA(IntPtr.Zero, "Hello", "Test", 0);
```

### 6.2 常见关键词

| 关键词 | 含义 |
|---|---|
| `[DllImport]` | 声明函数来自哪个 DLL |
| `extern` | 方法实现在外部 |
| `static` | 不需要实例即可调用 |
| `IntPtr` | 指针或句柄 |
| `Marshal` | 托管/非托管数据转换 |
| `SetLastError` | 保存 Windows API 错误码 |
| `CharSet` | 字符串编码，Ansi/Unicode/Auto |

### 6.3 为什么 `IntPtr` 很重要

指针大小和进程位数有关：

| 进程位数 | 指针大小 |
|---|---|
| 32 位 | 4 字节 |
| 64 位 | 8 字节 |

`IntPtr` 会自动适配当前进程位数，所以在 C# P/Invoke 中非常常见。

VBA 中对应概念是 `LongPtr`。

### 6.4 Marshal.Copy 是什么

`Marshal.Copy` 常用于把托管字节数组复制到非托管内存：

```csharp
byte[] data = new byte[] { 0x90, 0x90, 0xC3 };
IntPtr addr = Marshal.AllocHGlobal(data.Length);
Marshal.Copy(data, 0, addr, data.Length);
```

在 shellcode runner 里，常见流程是：

```text
byte[] shellcode
  -> VirtualAlloc 分配非托管内存
  -> Marshal.Copy 写入 shellcode
  -> CreateThread 从该地址执行
```

## 七、基础编程概念速通

### 7.1 变量和类型

```csharp
int number = 42;
string text = "hello";
bool ok = true;
byte[] bytes = new byte[] { 0x41, 0x42 };
IntPtr ptr = IntPtr.Zero;
```

OSEP 中常见类型：

| 类型 | 用途 |
|---|---|
| `byte[]` | shellcode、二进制程序集、加密数据 |
| `string` | 命令、URL、注册表路径 |
| `bool` | API 成功/失败 |
| `int/uint` | 标志位、PID、长度 |
| `IntPtr` | 地址、句柄、函数指针 |

### 7.2 控制流

```csharp
if (addr == IntPtr.Zero)
{
    Console.WriteLine("failed");
}
else
{
    Console.WriteLine("success");
}
```

循环：

```csharp
for (int i = 0; i < bytes.Length; i++)
{
    Console.WriteLine(bytes[i]);
}
```

异常处理：

```csharp
try
{
    // 可能失败的操作
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

### 7.3 方法

```csharp
public static int Add(int a, int b)
{
    return a + b;
}
```

方法签名里要看：

| 部分 | 例子 | 含义 |
|---|---|---|
| 访问修饰符 | `public` | 谁能访问 |
| 静态标记 | `static` | 是否属于类本身 |
| 返回类型 | `int` | 返回什么 |
| 方法名 | `Add` | 调用名称 |
| 参数 | `int a, int b` | 需要输入什么 |

### 7.4 类和对象

```csharp
public class Runner
{
    private byte[] payload;

    public Runner(byte[] payload)
    {
        this.payload = payload;
    }

    public void Execute()
    {
        Console.WriteLine(payload.Length);
    }
}
```

初学者要记住：

```text
类 = 模板
对象 = 根据模板创建出的实例
构造函数 = 创建对象时初始化数据
方法 = 对象或类能做的动作
字段/属性 = 对象保存的数据
```

详细 OOP 放在 [03-面向对象编程基础](/blog/osep/02-操作系统与编程理论/03-面向对象编程基础/)。

## 九、初学者常见误区

| 误区 | 正确理解 |
|---|---|
| “C# 是高级语言，所以不能做底层操作” | C# 可以通过 P/Invoke 调 Win32 API |
| “PowerShell 只是命令行” | PowerShell 能直接使用大量 .NET 能力 |
| “Base64 就是加密” | Base64 只是编码，不提供真正隐藏 |
| “Windows 是 x64 就一定用 x64 payload” | 要看当前进程位数，例如 Office 可能是 32 位 |
| “shellcode 是脚本” | shellcode 是 CPU 可执行的机器码字节 |
| “反射就是免杀” | 反射是动态调用机制，不等于天然免杀 |
| “API 返回 0 就是网络问题” | 可能是参数、权限、位数、内存保护错误 |

## 十、建议实验顺序

### 实验 1：理解语言执行方式

1. 写一个 Python `print("hello")`。
2. 写一个 C# `Console.WriteLine("hello")`。
3. 编译 C#，观察生成的 `.exe`。
4. 思考：Python 文件和 C# `.exe` 的执行依赖有什么不同？

### 实验 2：观察 .NET 程序集

1. 编译一个简单 C# 程序。
2. 用 dnSpy 或 ILSpy 打开。
3. 找到类、方法、字符串。
4. 理解 `.exe` 里不只是机器码，还有元数据。

### 实验 3：P/Invoke 调用 MessageBox

```csharp
using System;
using System.Runtime.InteropServices;

class Program
{
    [DllImport("user32.dll", CharSet = CharSet.Ansi)]
    static extern int MessageBoxA(IntPtr hWnd, string text, string caption, uint type);

    static void Main()
    {
        MessageBoxA(IntPtr.Zero, "Hello OSEP", "PInvoke", 0);
    }
}
```

### 实验 4：判断进程位数

```csharp
using System;

class Program
{
    static void Main()
    {
        Console.WriteLine(Environment.Is64BitOperatingSystem);
        Console.WriteLine(Environment.Is64BitProcess);
        Console.WriteLine(IntPtr.Size);
    }
}
```

对应 PowerShell：

```powershell
[Environment]::Is64BitOperatingSystem
[Environment]::Is64BitProcess
[IntPtr]::Size
```
