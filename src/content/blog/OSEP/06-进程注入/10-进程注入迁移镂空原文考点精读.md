# 进程注入、迁移与镂空原文考点精读

## 这篇文章解决什么问题

这篇文章对应参考资料目录中的 `6：进程注入迁移挖空.html`，核心不是让你背几个 API 名字，而是让你真正理解：为什么攻击者拿到代码执行以后，还要把代码放进别的进程里运行。

用最直白的话说，进程注入解决的是“我不想让 payload 只待在当前这个脆弱、显眼、容易退出的位置”。进程迁移解决的是“我已经执行了，但当前进程不稳定或权限/网络/架构不合适，需要换一个更适合长期操作的宿主”。进程镂空解决的是“我希望外表看起来像一个正常进程，但内部执行的是我准备好的代码”。

如果你之前学 Office 宏、JScript、Reflective PowerShell 时感觉像是在学“怎么让代码跑起来”，那这一章开始学习的是“代码跑起来以后，怎样在内存里选择位置、伪装身份、提高稳定性、绕开静态检查和一部分行为关注点”。

## 原文主题映射

本篇根据 OSEP/PEN-300 客户端执行链后续的进程注入主题整理，重点覆盖以下原文考点方向：

- Process Injection：把代码写入另一个进程的内存，并让那个进程执行它。
- Process Migration：把会话或执行能力从当前进程转移到另一个更稳定或更合适的进程。
- Process Hollowing：先创建一个看似正常的进程，再把它原本的代码“掏空/替换”为攻击者代码。
- Win32 API 与内存执行：理解打开进程、申请内存、写入内存、修改权限、创建远程线程等动作为什么按这个顺序出现。
- 稳定性与 OPSEC：注入对象、架构位数、权限级别、会话上下文、防护产品行为检测都会影响成败。

你读原文时可以把这一章放在攻击链中的这个位置：

```text
初始执行
  -> 宏 / JScript / PowerShell / C# 让代码第一次运行
  -> 进程注入/迁移/镂空让代码换位置、换宿主、换表现形式
  -> 凭证获取 / 横向移动 / 域内攻击继续展开
```

## 前置知识

读这一章前，不需要你已经是 Windows 内核专家，但至少要能接受下面几个概念：

- 进程：正在运行的程序实例，比如 `notepad.exe`、`explorer.exe`、`powershell.exe`。
- 线程：进程里真正被 CPU 调度执行的路线。一个进程可以有多个线程。
- 虚拟内存：每个进程都有自己的内存空间，不能随便读写别的进程，需要权限和 API。
- 句柄：Windows 给你的一张“操作某个对象的凭证”，比如进程句柄、线程句柄、文件句柄。
- 架构位数：32 位进程和 64 位进程不是一回事。32 位 payload 注入 64 位目标时经常会出问题。
- Token/权限：你能不能打开目标进程，取决于当前用户权限、完整性级别、目标进程保护状态。

## 一句话模型

进程注入可以先记成四个动作：

```text
找目标进程 -> 往目标内存放代码 -> 让目标内存变成可执行 -> 启动线程去跑
```

更贴近 Win32 API 的版本通常长这样：

```text
OpenProcess
  -> VirtualAllocEx
  -> WriteProcessMemory
  -> VirtualProtectEx
  -> CreateRemoteThread
```

不要一开始就死背函数名。先理解每一步在干什么：

- `OpenProcess`：我要先拿到目标进程的操作权限。
- `VirtualAllocEx`：我要在“别人的进程内存里”申请一块空间。
- `WriteProcessMemory`：我要把 shellcode 或 loader 写进去。
- `VirtualProtectEx`：我要让那块内存具备合适权限，常见是可执行。
- `CreateRemoteThread`：我要让目标进程开一条线程，从我写入的位置开始执行。

## 为什么不能只在当前进程里跑

初学者很容易困惑：既然宏或 JScript 已经能执行代码，为什么还要折腾注入？

原因通常有五类。

第一，当前进程可能太短命。比如 Office 文档关闭以后，`WINWORD.EXE` 里的代码就没了。迁移到更稳定的进程，可以让会话更不容易被用户一个关闭动作带走。

第二，当前进程太显眼。`powershell.exe`、`wscript.exe`、`cscript.exe` 这些进程在安全产品眼里常常比较敏感。把后续能力移到更常见的进程里，可能降低被简单规则命中的概率。

第三，当前进程权限不合适。你拿到的执行点可能权限有限，或者处于不方便访问目标资源的上下文。迁移不等于自动提权，但它可以让你利用更合适的会话、桌面、网络或进程环境。

第四，当前进程架构不合适。32 位 Office 里跑 32 位 payload，如果你后面要操作 64 位目标或加载 64 位组件，就会遇到位数不匹配。

第五，后续攻击链需要更灵活的内存执行。反射加载、shellcode runner、C# loader、进程镂空，本质上都在围绕“如何让代码不落地或少落地，并在合适位置运行”展开。

## 进程注入：把代码塞进另一个进程

进程注入的重点不是“神奇隐藏”，而是“跨进程内存操作”。你要对另一个进程做三件事：拿权限、写内存、触发执行。

典型链条如下：

```text
1. 选择目标进程
2. 获取目标进程 PID
3. 用足够权限打开目标进程
4. 在目标进程内存中申请空间
5. 写入 shellcode 或 loader
6. 设置内存权限
7. 创建远程线程或劫持已有执行流
8. 验证 payload 是否回连、是否执行、目标进程是否崩溃
```

### 选择目标进程时看什么

目标进程不是随便挑的。考试和实验里，你应该优先思考这些问题：

- 稳定性：这个进程会不会很快退出。
- 用户上下文：它属于哪个用户，能不能访问你需要的资源。
- 架构位数：目标进程和 payload 都是 32 位还是 64 位。
- 权限级别：你当前权限能不能打开它并写内存。
- 噪音程度：注入这个进程是否非常反常。
- 防护关注度：浏览器、Office、PowerShell、系统服务的检测规则都可能不同。

一个常见误区是觉得“注入到系统进程最隐蔽”。实际上系统进程权限更高、防护更多、崩溃影响更大，失败时也更明显。考试里更重要的是稳定、可解释、可复现，而不是追求听起来酷的目标。

### 成功信号

进程注入成功通常会出现这些信号：

- handler 收到新的会话或原会话变稳定。
- 目标进程仍然存活，没有立刻崩溃。
- 日志或调试输出显示远程线程创建成功。
- 目标内存区域可见，权限与预期一致。
- 后续命令在目标进程上下文中执行。

### 失败信号

失败时不要只盯着“没有回连”。你要分层排查：

- 打不开进程：权限不够、PID 错、目标进程退出、完整性级别不匹配。
- 申请不了内存：权限不足、参数错误、目标架构不匹配。
- 写不进去：句柄权限不够、地址无效、payload 太大。
- 跑不起来：内存不可执行、入口地址错、shellcode 架构不匹配。
- 一跑就崩：payload 与目标环境不兼容，或破坏了目标进程状态。
- 有执行但无回连：网络过滤、监听器错误、payload 配置错误、DNS/代理问题。

## 进程迁移：把会话搬到更合适的宿主

进程迁移可以理解为“不是重新打一次靶，而是把已经拿到的执行能力搬家”。

它常见于这些场景：

- 你从 Office 宏拿到会话，但用户可能随时关闭 Word。
- 你从脚本宿主拿到会话，但 `wscript.exe` 或 `powershell.exe` 很显眼。
- 当前会话不稳定，经常断。
- 你需要进入目标用户长期运行的进程环境。
- 你需要从 32 位进程迁移到 64 位进程，或者反过来处理兼容性。

### 迁移不是提权

这句话很重要：迁移不等于提权。

如果你当前只有普通用户权限，通常不能随便迁移到高权限系统进程。迁移更多是在已有权限范围内选择更合适的载体。你可以把它想成“换房间”，不是“自动升楼层”。

### 迁移时最容易踩的坑

第一，位数不匹配。32 位会话迁移到 64 位进程时，需要对应架构的 payload 或迁移机制。

第二，选了会退出的进程。比如临时启动的程序、安装器、某些短生命周期子进程。

第三，迁移到关键进程导致崩溃。考试里把稳定进程弄崩，可能直接断链或影响后续利用。

第四，误以为迁移后所有限制都消失。网络过滤、权限不足、凭证不可用、防护策略仍然可能存在。

第五，报告里只写“migrate 成功”。更好的写法是说明为什么迁移、迁移前后 PID/进程名/架构/用户上下文有什么变化。

## 进程镂空：外壳是真的，里面换了

Process Hollowing 也叫进程镂空。它的名字很形象：先启动一个正常程序，让系统看见一个合法进程；然后把这个进程原本映像中的关键内容替换掉，让它执行攻击者准备的代码。

初学者可以把它拆成下面的流程：

```text
1. 以挂起状态创建一个正常进程
2. 获取它的主线程和进程上下文
3. 解除或替换原始映像中的关键内存区域
4. 把恶意映像或 shellcode 写入目标进程
5. 修正入口点、PEB 或线程上下文
6. 恢复主线程执行
```

它和普通远程线程注入最大的区别是：普通注入更像“给别人家里塞一段新代码，再开一条新线程”；镂空更像“房子的门牌和外观还在，但里面的结构被换掉，然后继续从主入口走”。

## 为什么原文要反复强调 API 顺序

这些 API 的顺序不是仪式感，而是 Windows 对内存和进程安全边界的基本要求。

你不能写入一个你没有权限操作的进程，所以要先拿句柄。你不能把代码写到不存在的地址，所以要先申请内存。你不能让 CPU 执行不可执行的内存，所以要处理内存保护属性。你不能指望代码自己运行，所以要创建线程或改变已有线程的执行位置。

这也是考试排错的核心：每一步都有一个明确目的，所以每一步都可以单独验证。

## 考试怎么用

考试中不要为了“使用进程注入而使用进程注入”。它应该服务于目标。

适合使用的情况：

- 初始会话来自 Office、浏览器、脚本宿主，明显不稳定。
- 当前进程退出风险高。
- 你需要更合适的 64 位进程环境。
- 后续模块需要长期会话。
- 防护或日志让当前宿主非常扎眼。

不适合硬上的情况：

- 当前会话已经稳定，且你没有必要增加风险。
- 目标进程权限、架构、生命周期都没确认。
- 你不确定 payload 与目标架构是否匹配。
- 你没有回连备用方案。
- 你无法解释迁移或注入为什么必要。

报告里应该记录：

- 注入或迁移前的进程名、PID、用户、架构。
- 目标进程选择理由。
- 使用的 payload 类型和监听器配置。
- 成功后的新 PID、进程名、权限上下文。
- 遇到的失败原因与修正方式。

## 常见误区

- 误区一：进程注入等于免杀。注入只是执行位置变化，不等于绕过所有检测。
- 误区二：迁移等于提权。迁移通常不会突破当前权限边界。
- 误区三：系统进程一定更好。越敏感的进程越容易失败，也越容易造成严重副作用。
- 误区四：只要 shellcode 对就一定能跑。架构、内存权限、调用约定、目标进程状态都会影响结果。
- 误区五：没有回连就是注入失败。也可能是网络过滤、监听器错误、代理、DNS、payload 参数问题。
- 误区六：镂空只是高级版注入。它涉及 PE 映像、线程上下文和入口点修正，排错维度更多。

## 排错顺序

遇到失败时，按这个顺序排，比盲目换 payload 更有效：

```text
1. 我当前是什么用户和完整性级别？
2. 目标进程是否存在、是否稳定、是否同架构？
3. 我是否拿到了需要的进程权限？
4. 远程内存是否申请成功？
5. 写入字节数是否符合预期？
6. 内存权限是否允许执行？
7. 线程是否创建或恢复成功？
8. payload 是否与监听器和网络路径匹配？
9. 防护产品是否拦截了某一步行为？
10. 是否有更安全的目标进程或更简单的执行路径？
```

如果你只能记一条排错原则，就记这个：先确认“代码有没有被放进去”，再确认“代码有没有被允许执行”，最后确认“执行后有没有网络回连”。

## 读不懂时回退到哪里

如果你卡在 API 名字上，先回退到“进程、线程、内存、句柄”四个概念，不要强行背代码。

如果你卡在 shellcode 为什么能执行，回退到 Office 或 PowerShell shellcode runner 章节，重新理解“申请内存、复制字节、跳转执行”。

如果你卡在为什么要迁移，回到钓鱼章节想象一个现实场景：用户关掉 Word，你的会话会怎样。

如果你卡在镂空，先不要急着看完整实现，先画出“创建挂起进程 -> 替换内容 -> 改入口 -> 恢复执行”这四步。

如果你卡在考试怎么选，问自己两个问题：

- 我迁移或注入是为了稳定、架构、隐蔽、权限上下文，还是只是因为教程这么写？
- 如果这一步失败，我有没有不破坏当前会话的回退方案？
