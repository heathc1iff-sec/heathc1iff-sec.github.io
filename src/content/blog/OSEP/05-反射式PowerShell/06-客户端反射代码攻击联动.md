# 客户端反射代码攻击联动

## 对应课程小节

`web` 目录中的 `5：客户端反射代码攻击.html` 对应 `Reflective Code Execution in Client Side Attacks`，核心是把前面学过的反射式 PowerShell、反射式 C# 和客户端投递方式组合起来。

这章不是新 API，而是一个组合题：入口来自 Office/JScript，执行能力来自 .NET 反射，目标是减少落地文件和编译痕迹。

---

## 一句话理解

客户端反射代码攻击就是：

```
钓鱼载体负责启动
    -> 内存加载脚本或程序集
    -> 反射解析 API 或 .NET 类型
    -> 分配内存 / 加载程序集 / 执行入口点
    -> 尽量避免 Add-Type、csc.exe、明显文件落地
```

---

## 和前置章节的关系

| 前置能力 | 在本章里的作用 |
|---|---|
| Office/VBA | 启动 PowerShell、WMI、mshta、wscript 或加载内嵌脚本 |
| JScript/WSH | 作为脚本型投递器，调用 COM、运行 .NET 载荷或解码内容 |
| 反射式 PowerShell | 在内存中定位 Win32 API 并创建委托调用 |
| C# Shellcode Runner | 提供类型、委托、P/Invoke、Assembly.Load 的思路 |
| 杀软绕过 | 字符串拆分、编码、延迟、环境检查、行为解链 |
| AppLocker/CLM | 决定是否能用 PowerShell、JScript、MSHTA 或第三方加载器 |

---

## 组合路线

| 路线 | 入口 | 反射执行点 | 优点 | 卡点 |
|---|---|---|---|---|
| VBA -> PowerShell 反射 | Office 宏 | PowerShell 查找 API 并调用委托 | 快速、课程主线清晰 | AMSI、PowerShell 日志、CLM |
| VBA -> WMI -> PowerShell | Office 宏 | WMI 解链后执行反射脚本 | 降低父子进程关联 | WMI 事件明显、脚本仍可能被 AMSI |
| JScript -> DotNetToJScript | WSH | 反序列化或加载 .NET 程序集 | 不依赖 PowerShell 入口 | .NET 版本、脚本规则、签名检测 |
| JScript -> C# Assembly.Load | WSH/MSHTA | 内存加载程序集入口点 | 可复用 C# 逻辑 | AMSI/Script Host 检测、架构匹配 |
| Office -> HTML Smuggling -> Loader | 浏览器/Office | 载荷由用户侧保存或运行 | 初始投递更灵活 | 用户交互、下载标记、SmartScreen |

## 入口层、加载层、能力层对照

| 层级 | 可选项 | 选择依据 | 失败后替换方向 |
|---|---|---|---|
| 入口层 | Office 宏、JScript、HTA、HTML Smuggling、日历凭证诱导 | 用户交互、文件类型限制、脚本宿主是否可用 | 换投递载体或转凭证入口 |
| 加载层 | PowerShell 反射、DotNetToJScript、Assembly.Load、WMI 解链 | AMSI/CLM、AppLocker、.NET 版本、代理 | 换宿主、预编译、可信加载点 |
| 能力层 | Shellcode Runner、主机枚举、下载器、凭证访问、横向组件 | 当前目标是 foothold、枚举还是移动 | 降级为最小命令或换模块 |

考试里先证明入口层能执行，再证明加载层能稳定运行，最后再换成完整能力层。不要一开始就把三层混成一个大 payload，否则失败时很难判断是入口、加载还是能力本身出了问题。

---

## 设计载荷时先问 8 个问题

1. 目标能不能运行 Office 宏或脚本宿主。
2. PowerShell 是否可用，是否是 Constrained Language Mode。
3. AMSI 是否会扫描当前路径。
4. AppLocker/WDAC 管的是 EXE、脚本、MSI 还是 DLL。
5. 载荷是否需要落地，落地路径是否可信。
6. 目标是 x86 还是 x64，Office 位数和系统位数是否一致。
7. 出网是否经过代理，SYSTEM 和普通用户的代理配置是否不同。
8. 失败时是否能从日志、错误码或父进程关系判断原因。

## 关键实现思路

### 1. 避免 Add-Type

`Add-Type` 会触发临时源码和编译链路，容易留下痕迹，也容易被策略限制。反射路线的重点是直接从已加载程序集里找可用类型，再构造委托调用目标函数。

复习时要能解释：

| 步骤 | 目的 |
|---|---|
| 枚举已加载程序集 | 找到 `System.dll` 等可用位置 |
| 定位 `UnsafeNativeMethods` | 复用框架内部的 API 查找能力 |
| 调用 `GetModuleHandle` / `GetProcAddress` | 获取 Win32 API 地址 |
| 动态创建委托类型 | 不写 C# 源码也能调用非托管函数 |
| `Marshal.GetDelegateForFunctionPointer` | 把函数地址转成可调用对象 |

### 2. C# 程序集内存加载

内存加载程序集的核心不是“神奇免杀”，而是减少写盘和编译动作：

```
字节数组
  -> 解码/解密
  -> Assembly.Load(byte[])
  -> 找到 Type 和 Method
  -> Invoke 入口方法
```

考试里要注意，`Assembly.Load` 加载的程序集仍然可能触发 AMSI、ETW、行为检测和 .NET 事件，所以它只是执行方式，不是免杀保证。

### 3. 载体与执行逻辑分离

好的考试模板应该分成三层：

| 层 | 职责 |
|---|---|
| 载体层 | VBA、JScript、HTA，只负责启动和解码 |
| 加载层 | 反射、Assembly.Load、代理配置、位数判断 |
| 能力层 | Shellcode Runner、枚举器、下载器、横向移动组件 |

这样某一层被拦时，只需要替换一层，不必整条链重写。

## 常见失败与排查

| 现象 | 优先排查 |
|---|---|
| Office 里没有回连 | 宏是否启用、位数是否匹配、代理、父进程是否启动 |
| PowerShell 报语法或类型错误 | CLM、反射类型名、.NET 版本、引号转义 |
| `AccessViolation` 或进程崩溃 | API 声明、指针大小、Shellcode 架构、内存保护 |
| JScript 加载 .NET 失败 | .NET 版本、COM/ActiveX 策略、脚本执行规则 |
| 本地测试成功目标失败 | 代理、位数、Defender、AppLocker、语言环境、路径权限 |
| 载荷落地后不能执行 | Mark-of-the-Web、签名规则、路径规则、文件扩展名规则 |

## VBA -> 反射 PowerShell 与 JScript -> 反射 C# 对比

| 维度 | VBA -> 反射 PowerShell | JScript -> 反射 C# |
|---|---|---|
| 父进程 | `WINWORD.EXE` / `EXCEL.EXE` | `wscript.exe` / `cscript.exe` / `mshta.exe` |
| AMSI 面 | PowerShell 内容更容易触发 AMSI | .NET/JScript 也可能触发，但规则不同 |
| CLM 面 | PowerShell 受 CLM 影响明显 | 取决于脚本宿主和 .NET 加载方式 |
| 位数要求 | Office 位数影响 API 声明和 shellcode | .NET 目标架构和宿主位数要匹配 |
| 代理问题 | 可用 `WebClient`/系统代理 | 需要显式处理下载器或内嵌载荷 |
| 落地风险 | 可纯内存，但宏文档本身敏感 | JScript/HTA 文件和 ActiveX 策略敏感 |
| 考试优先级 | Office 可用时优先，排错资料多 | 宏受限或要绕开 PowerShell 时优先 |

选择原则：PowerShell 可用就先用反射 PowerShell 快速验证；PowerShell/AMSI/CLM 卡住时，优先把加载层换成 DotNetToJScript 或内存加载 C#。

## 复习闭环

完成这个主题时，至少能手写下面这条逻辑：

```
如果 PowerShell 可用：
  优先尝试反射式 PowerShell，注意 AMSI/CLM。

如果 PowerShell 被限制：
  看 JScript/MSHTA/XSL/InstallUtil/Workflow Compiler 是否可用。

如果脚本被限制但 DLL 可用：
  看可信目录、DLL 搜索顺序、第三方签名程序加载点。

如果只能运行白名单程序：
  回到 AppLocker 章节，找可控输入到可信执行器的路径。
```

## 考试排错闭环

| 失败位置 | 先验证 | 再切换 |
|---|---|---|
| 用户打开后无任何反应 | 宏/脚本是否启动、文件是否被保护视图或策略阻止 | 换入口层：JScript、HTA、HTML Smuggling |
| 能执行命令但加载器失败 | PowerShell 语言模式、AMSI 报错、.NET 版本、位数 | 换加载层：DotNetToJScript、Assembly.Load、可信加载点 |
| 加载器运行但无回连 | 代理、TLS、监听器、payload 架构、父进程链 | 换能力层或先做最小出网探测 |
| 本地成功目标失败 | AppLocker/WDAC、Defender 签名、Office 位数、系统区域设置 | 进入 AV/AMSI/AppLocker 决策树 |
