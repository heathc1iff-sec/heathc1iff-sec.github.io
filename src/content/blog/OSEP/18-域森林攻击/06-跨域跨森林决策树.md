# 跨域跨森林决策树

## 这篇文档解决什么问题

跨域、跨森林是 AD 后段最容易混乱的部分。初学者常见误区是：看到 trust 就以为能访问，看到 SID History 就以为能跨所有森林，看到 SQL Link 就不知道它和 AD 信任有什么关系。

这篇文档按 OSEP 复习角度回答：

```text
我在哪个域
  -> 我能看到哪些信任
  -> 信任方向代表谁能访问谁
  -> 是否同森林
  -> SID Filtering/选择性认证是否限制
  -> 我手里有什么高价值材料
  -> 选择 Extra SID、Trust Key、SQL Link、ADCS、委派或凭证复用
```

只用于授权实验环境和考试备考。

---

## 先分清三个层级

| 层级 | 你要问的问题 |
|---|---|
| 域内 | 我能不能在当前域提权到 Domain Admin 或 DCSync |
| 森林内跨域 | 我能不能从子域影响父域或同森林其他域 |
| 森林间 | 我能不能通过显式 trust、凭证、SQL Link、ADCS 或服务配置跨到另一个森林 |

不要一开始就追跨森林。先确认当前域是否已经有足够权限、是否能拿到 krbtgt、是否能枚举 trust。

---

## 信任方向怎么读

信任方向容易反。用一句话记：

```text
A trusts B = A 信任 B 的用户来访问 A 的资源
```

| 枚举结果 | 人话解释 | 你要验证 |
|---|---|---|
| Inbound | 对方信任我方 | 我方身份可能访问对方资源 |
| Outbound | 我方信任对方 | 对方身份可能访问我方资源 |
| Bidirectional | 双向互信 | 两边身份都可能访问对方资源 |

不同工具对 inbound/outbound 的表述可能以“当前域”为视角。报告里一定要写清楚“从哪个域看”。

## 正序枚举流程

### 第 1 步：确认当前域和 SID

```powershell
whoami /fqdn
whoami /user
nltest /dsgetdc:domain.local
```

PowerView：

```powershell
Get-Domain
Get-DomainSID
```

记录：

| 项目 | 记录 |
|---|---|
| 当前域 |  |
| 当前域 SID |  |
| 当前用户 |  |
| 当前权限 |  |
| DC |  |

### 第 2 步：枚举森林和信任

```powershell
Get-DomainTrust
Get-ForestTrust
Get-ForestDomain
```

系统命令：

```cmd
nltest /domain_trusts
```

记录：

| 目标域/森林 | 信任方向 | 信任类型 | 是否传递 | 是否同森林 | 备注 |
|---|---|---|---|---|---|
|  |  |  |  |  |  |

### 第 3 步：检查是否能解析和访问对方

```powershell
Resolve-DnsName dc.other.local
Test-NetConnection dc.other.local -Port 88
Test-NetConnection dc.other.local -Port 389
Test-NetConnection dc.other.local -Port 445
```

跨域失败很多时候不是权限，而是 DNS、路由、端口。

### 第 4 步：判断你手里有什么材料

| 材料 | 可能路线 |
|---|---|
| 子域 DA + 子域 krbtgt | Extra SID 到森林根 |
| Trust Key | 构造跨域信任票据 |
| 对方域凭证 | 直接枚举对方域 |
| SQL Link | 绕过 AD 直觉的跨边界 SQL 身份链 |
| 机器账户/服务账户 | 委派、RBCD、ADCS、LDAP |
| ADCS 证书 | PKINIT/Schannel 跨边界访问 |

## 决策树

```text
发现 trust
  |
  +-- 是否同一森林？
  |     |
  |     +-- 是：看父子域、Enterprise Admins、Extra SID、krbtgt
  |     |
  |     +-- 否：看 SID Filtering、选择性认证、DNS/端口、可用凭证
  |
  +-- 当前是否已控制某个域？
  |     |
  |     +-- 有子域 DA/krbtgt：考虑 Extra SID / SID History
  |     +-- 只有普通用户：先枚举对方可访问资源和凭证复用
  |
  +-- 是否有跨边界服务链？
        |
        +-- SQL Link：走 MSSQL 链路
        +-- ADCS 暴露：走证书路径
        +-- 委派/打印机/强制认证：走 Kerberos/Relay 路径
```

---

## Extra SID / SID History 路线

### 使用场景

- 同森林父子域。
- 你控制子域高权限。
- 能拿到子域 `krbtgt`。
- 能获取父域 SID。

### 核心逻辑

```text
用子域 krbtgt 签一个子域票据
  -> 在 Extra SID 中加入父域高权限组 SID
  -> 访问父域资源
```

### 常见卡点

| 现象 | 排查 |
|---|---|
| 票据能注入但访问失败 | 父域 SID/RID 错、目标服务名错 |
| 跨森林不生效 | SID Filtering |
| 找不到父域 SID | DNS/LDAP/信任枚举问题 |
| 访问 SMB 失败 | CIFS SPN、目标 FQDN、票据服务类型 |

---

## 跨森林路线

跨森林要更保守地判断。存在 trust 不代表你能随意访问。

优先检查：

| 条件 | 说明 |
|---|---|
| DNS | 能否解析对方域和 DC |
| Kerberos/LDAP/SMB 端口 | 网络是否允许 |
| SID Filtering | 是否过滤外来高权限 SID |
| Selective Authentication | 是否需要显式允许登录目标主机 |
| 可用凭证 | 当前身份是否在对方有权限 |
| 服务链 | SQL Link、ADCS、委派是否跨边界 |

### SQL Link 跨森林

SQL Link 可能已经配置好跨森林身份映射。你不一定能用 AD 凭证直接访问对方域，但 SQL Server 之间能互相查询。

## 排错表

| 现象 | 优先排查 |
|---|---|
| 有 trust 但不能枚举 | 信任方向、DNS、LDAP、防火墙、权限 |
| 跨域票据不能用 | SID、RID、SPN、时间、目标服务 |
| 跨森林 Extra SID 失败 | SID Filtering，属正常限制 |
| 能解析但不能访问 | 选择性认证、目标 ACL、端口 |
| SQL Link 可用但 AD 不通 | SQL 身份链和 AD 信任不是一回事 |
| 对方域凭证登录失败 | UPN/域名格式、账号限制、时间、策略 |

---

## OSEP 考试相关重点

1. 信任方向、类型、是否同森林必须写清楚。
2. Extra SID 更偏森林内父子域，不要套到所有跨森林场景。
3. 跨森林常被 SID Filtering 和选择性认证限制，失败本身可能是正确结论。
4. SQL Link、ADCS、委派可能绕过你对 trust 的直觉，是重点检查项。
5. 报告里要记录当前域、目标域、信任方向、材料、票据/访问验证结果。
