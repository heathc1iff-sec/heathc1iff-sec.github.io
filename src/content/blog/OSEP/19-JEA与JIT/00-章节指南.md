# JEA 与 JIT 章节指南

## 这个目录是干什么的

JEA 和 JIT 对应参考资料 `18：ad内部攻击向量（JEA以及JIT策略）.html` 的后半部分：`Just Enough Administration (JEA)` 和 `Just-In-Time (JIT) Access`。同一个 HTML 前半部分讲 Kerberos 委派，那部分主章节是 `17-Kerberos委派`，本目录只负责 JEA/JIT。

这两个概念都来自企业的权限收敛思路：

| 机制 | 企业想解决什么问题 | 攻击视角要看什么 |
|---|---|---|
| JEA | 不给完整管理员，只给一个受限 PowerShell 管理端点 | 端点暴露了哪些命令、这些命令能否读写文件或启动程序 |
| JIT | 不长期保留高权限，只在短时间窗口内临时授权 | 临时组成员关系能否变成本地管理员、服务访问或横向移动能力 |

初学者不要把它们理解成“一个命令就能利用”的漏洞。JEA/JIT 更像权限系统里的缝隙：你要先看它给了什么管理能力，再判断这个能力是否能转成文件控制、命令执行、凭证访问或横向移动。

---

## 和 Kerberos 委派的边界

`18：ad内部攻击向量（JEA以及JIT策略）.html` 同时包含 Kerberos Delegation、JEA 和 JIT。复习时一定要拆开：

| 看到的线索 | 主章节 | 你在判断什么 |
|---|---|---|
| `TrustedForDelegation`、`msDS-AllowedToDelegateTo` | `17-Kerberos委派` | 服务能否代表用户访问其他服务 |
| `msDS-AllowedToActOnBehalfOfOtherIdentity` | `17-Kerberos委派` | 是否存在 RBCD 路径 |
| `Enter-PSSession -ConfigurationName ...` | `19-JEA与JIT` | 是否存在受限 PowerShell 管理端点 |
| `.pssc`、`.psrc`、Role Capability | `19-JEA与JIT` | 端点允许哪些命令和参数 |
| 临时管理员、PAM、JIT Web 入口、TTL 组成员 | `19-JEA与JIT` | 临时权限能否在有效期内转成访问能力 |

考试可用的一句话：委派解决“服务能代表谁访问哪里”，JEA/JIT 解决“受限或临时管理入口能不能变成真实权限”。

## 先建立的核心模型

### JEA 的判断模型

```text
发现端点
  -> 能否连接
  -> 连接后是什么身份
  -> 允许哪些命令
  -> 命令参数是否受限
  -> 能否读敏感文件
  -> 能否写可执行位置
  -> 能否启动程序、服务或脚本
  -> 能否拿到凭证或本地管理员能力
```

JEA 的重点不是“端点存在”，而是端点里有哪些命令被错误暴露。比如 `Copy-Item`、`Start-Process`、`Restart-Service`、`Invoke-*`、可写路径和不严格的参数验证，才是你要重点分析的地方。

### JIT 的判断模型

```text
发现 JIT/PAM 线索
  -> 找到请求入口或相关组
  -> 判断谁能请求、谁审批、加到哪个目标组
  -> 确认临时权限有效期
  -> 刷新或重新获取 Kerberos 票据
  -> 验证目标主机或服务权限
  -> 在时间窗口内完成凭证、横向或证据收集
```

JIT 的重点是“临时权限是否真的进入你的访问令牌/票据”。很多失败不是权限没给，而是你还在用旧票据、旧会话或错误目标。

## JEA 初学者阅读法

第一次读 JEA 时，按下面 6 个问题做笔记：

| 问题 | 说明 | 推荐文件 |
|---|---|---|
| 端点在哪里 | 端点名、目标主机、连接方式 | `01-JEA-JIT利用.md` |
| 谁能连接 | 当前用户、组成员、端点访问控制 | `03-JEA-JIT初学者决策树.md` |
| 连接后是谁 | RunAs、虚拟账户、普通用户还是管理员上下文 | `01-JEA-JIT利用.md` |
| 允许哪些命令 | `Get-Command`、可见函数、别名、外部程序 | `01-JEA-JIT利用.md` |
| 参数是否受限 | 是否限制路径、服务名、目标主机、文件扩展名 | `02-考试速查与排错.md` |
| 能转成什么 | 读文件、写文件、启动进程、重启服务、复制 DLL、读取凭证 | `03-JEA-JIT初学者决策树.md` |

JEA 里最容易漏的是参数边界。一个命令本身看起来普通，但如果路径、目标服务或参数没有限制，就可能从“管理操作”变成“任意文件读写或命令执行”。

## JIT 初学者阅读法

JIT 要按“组”和“时间窗口”来读：

| 问题 | 你要确认什么 |
|---|---|
| 请求入口在哪里 | Web 入口、管理端点、脚本、已有文档或浏览器历史 |
| 我能请求什么组 | 请求组、审批组、目标权限组之间的关系 |
| 目标组能做什么 | 本地管理员、远程管理、服务管理、文件共享、数据库权限 |
| 权限多久有效 | TTL、审批窗口、是否需要重新登录或刷新票据 |
| 权限是否已生效 | `whoami /groups`、Kerberos 票据、目标主机访问验证 |
| 失效前做什么 | 凭证收集、横向移动、证据截图、继续枚举 |

如果 JIT 申请成功后还是访问失败，优先想三件事：你有没有刷新票据，你访问的是不是目标组真正管的主机，你使用的协议是否读取了新的组成员关系。

## 失败现象索引

| 现象 | 可能原因 | 优先看 |
|---|---|---|
| 找不到 JEA 端点 | 没有枚举到 PowerShell 历史、WinRM、注册端点或文档线索 | `01-JEA-JIT利用.md` |
| 能看到端点但连不上 | 当前用户无权限、WinRM 不通、认证方式不对 | `02-考试速查与排错.md` |
| 连接后命令很少 | Role Capability 限制严格，先分析参数和输出能力 | `03-JEA-JIT初学者决策树.md` |
| `Copy-Item` 不能复制敏感文件 | 路径限制、RunAs 权限不足、目标路径不可写 | `02-考试速查与排错.md` |
| 能写文件但没有执行 | 需要服务重启、脚本宿主、计划任务或可加载位置 | `09-应用白名单绕过`、`11-Linux后渗透` |
| JIT 申请成功但权限没变 | 旧票据或旧会话，未刷新 Kerberos，目标组不是本机管理员 | `03-JEA-JIT初学者决策树.md` |
| 有临时权限但无法横向 | 端口、协议、UAC、远程管理策略或本地组范围不匹配 | `14-横向移动` |
| 不知道该归到哪章 | 发现 S4U/RBCD 回委派；发现端点/临时组留在本章 | 本文件边界表 |

## 学习目标

学完后你应该能回答：

1. JEA 和 JIT 分别解决什么管理问题。
2. 为什么 JEA 的重点是端点、RunAs、Role Capability、命令参数限制。
3. 为什么 `Copy-Item`、`Start-Process`、`Restart-Service`、`Invoke-*` 这类能力值得重点检查。
4. JIT 的请求组、审批组、目标权限组之间是什么关系。
5. 为什么 JIT 权限申请成功后还要刷新 Kerberos 票据或重开会话。
6. 如何判断失败是端点权限问题、命令限制问题、票据缓存问题，还是目标协议问题。
7. 如何把 JEA/JIT 的影响写成“从受限管理能力到真实访问能力”的报告证据。

## 完成标准

| 检查项 | 合格标准 |
|---|---|
| 边界 | 能把 Kerberos 委派、JEA、JIT 分开复盘 |
| JEA 枚举 | 能找到端点、连接端点、列出可用命令和当前身份 |
| JEA 判断 | 能分析命令参数是否能变成文件读写、执行或服务控制 |
| JIT 枚举 | 能解释请求组、审批组、目标权限组和有效期 |
| JIT 利用思路 | 能说明为什么要刷新票据，以及如何验证临时权限生效 |
| 排错 | 能按端点、身份、命令、票据、目标协议五层排查失败 |
| 报告 | 能证明权限扩大前后的差异和后续影响 |
